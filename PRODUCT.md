# PRODUCT

## 1. 文档目的

本文档用于约束当前产品阶段的事实边界，避免后续 `DESIGN.md`、UI 方案与页面文案把未确认内容误写成既定事实。

本文档仅记录当前已验证信息与明确待确认项，不扩展品牌、人群、商业模式、合规策略等未知内容。

## 2. 当前产品定位

当前产品是一个面向单用户使用的工作台式 MVP。

已确认能力范围包含：

- 基于手机号的账号体系
- 短信登录与密码登录
- 独立手机号注册页
- 个人声纹建声
- active voice 回放与作废
- 文本转语音（TTS）
- TTS 免费生成限制与一次性使用码
- 生成语音播放、下载与历史语音查看
- 账号信息展示与密码管理

首页同时承担未登录态登录入口与登录后工作台；注册页用于手机号注册；设置页用于账号信息展示与密码管理。

## 3. 一句话定位

一个以手机号账号登录为入口、完成个人建声、active voice 管理与文本转语音任务的单用户工作台式 MVP。

## 4. 当前可验证能力

以下内容可作为当前阶段的产品事实：

1. 产品存在手机号账号体系。
2. 用户可通过首页未登录态完成短信登录或密码登录。
3. 短信登录链路当前允许未注册手机号通过验证码后自动创建账号并进入系统。
4. 用户也可通过独立注册页完成手机号注册，并可选择设置密码。
5. 用户登录后进入首页工作台。
6. 工作台承载个人声纹建声、active voice 回放与作废、文本转语音、结果播放、结果下载与历史语音查看。
7. 设置页承担账号信息展示、首次设置密码与短信验证码修改密码职责。
8. 整体语气要求为专业、直接、低情绪、面向任务完成。

除以上内容外，本文档不推导任何额外能力。

## 5. 目标用户与使用前提

### 5.1 目标用户

当前仅能确认：

- 使用者是单个登录用户。
- 使用者进入系统的主要目标是完成个人建声、active voice 管理与文本转语音任务。
- 使用者通过中国大陆 11 位手机号完成账号进入链路。

### 5.2 使用前提

当前可确认的使用前提：

- 用户需要通过短信登录、密码登录或独立注册进入系统。
- 匿名用户可以在 Cookie 会话下使用首页工作台，但免费 TTS 机会用完后必须先登录。
- 建声依赖浏览器录音能力与麦克风权限。
- 建声录音当前要求不少于 5 秒。
- TTS 依赖当前可用的 active voice。
- TTS 免费生成单次输入不超过 30 字。
- 匿名用户和登录用户各有 1 次免费生成机会。
- 同一匿名 Cookie 会话登录后，已使用的匿名免费机会会迁移到注册账号，避免重复领取免费机会。
- 匿名用户不能使用使用码；匿名免费机会用完后必须先登录。
- 注册用户免费机会用完后，后续生成需要输入 6 位使用码。
- 注册用户可直接在前台看到通用使用码，默认通用使用码为 `123456`，可通过配置修改。
- 使用码按模块管理，当前只有 `VOICE_TO_TEXT` 模块。
- 前台只展示通用使用码，不展示非通用一次性使用码库存、消费记录或后台查询结果。
- 每条 TTS 任务需要记录权益来源：免费生成、通用使用码生成或非通用一次性使用码生成；使用码生成需记录本次输入的使用码，免费生成时使用码记录为空。
- 使用码一次性消费，消费后作废。
- 设置页用于查看账号信息、设置密码或修改密码。

除以上之外，不补充设备、行业、岗位、权限层级等假设。

## 6. 核心任务流

当前 MVP 的核心任务流按最小闭环理解为：

1. 用户通过首页短信登录进入系统；若手机号未注册，当前实现会在验证码通过后自动创建账号。
2. 用户也可以通过独立注册页完成手机号注册，再回到首页登录或进入工作台。
3. 用户进入首页工作台。
4. 用户在工作台完成个人声纹建声。
5. 用户回放或作废当前 active voice。
6. 用户在工作台输入文本并执行 TTS；免费机会用完后，注册用户需输入 6 位使用码。
7. 用户播放或下载生成语音，并可查看历史语音。
8. 用户如有账号信息查看或密码管理需求，进入设置页处理。

说明：

- 上述流程是当前产品骨架，不代表完整信息架构已最终确定。
- “短信登录即创建账号”是当前实现事实；是否长期作为产品策略仍需确认。
- 若后续出现更多流程节点，必须先有新证据，再更新本文档。

## 7. 页面职责

### 7.1 首页（未登录入口 + 登录后工作台）

- 未登录态提供短信登录与密码登录入口。
- 登录后作为默认主工作界面。
- 承载个人声纹建声任务。
- 承载 active voice 回放与作废。
- 承载文本转语音任务。
- 承载生成语音播放、下载与历史语音查看。
- 优先服务任务完成，不承担内容营销或信息浏览首页职责。

### 7.2 注册页

- 用于手机号注册。
- 支持手机号验证码注册，并允许注册时选择是否设置密码。
- 职责应聚焦账号创建，不应扩展为品牌介绍页或复杂转化页。

### 7.3 设置页

- 用于账号信息展示。
- 用于首次设置密码。
- 用于已设置密码用户通过短信验证码修改密码。
- 当前不应默认扩展为通用账户中心或多模块设置中心。

## 8. 设计原则

1. **任务优先**：页面组织必须服务“登录 / 注册 / 建声 / active voice 管理 / TTS / 下载 / 密码管理”主链路。
2. **单用户视角**：界面默认围绕单个用户操作，不引入团队、协作、审批等结构。
3. **MVP 收敛**：不为未确认能力预留过多显性 UI 入口。
4. **直接反馈**：关键状态、结果、错误信息应直接说明任务是否可继续。
5. **低干扰**：减少与任务无关的视觉元素、情绪化表达和装饰性叙事。
6. **事实优先**：所有页面结构与文案必须以已确认能力为上限，不得反向定义产品。

## 9. 文案与语气原则

当前文案与语气必须满足以下要求：

- 专业
- 直接
- 低情绪
- 面向任务完成

因此后续文案应避免：

- 夸张承诺
- 过强营销语气
- 拟人化陪伴表达
- 缺乏任务指向的情绪化欢迎语

更适合的表达方向：

- 明确动作：注册、登录、建声、回放、作废、生成、下载、管理
- 明确状态：成功、失败、处理中、待完成、无 active voice、暂无历史语音
- 明确结果：进入工作台、完成密码管理、完成语音生成、下载生成语音

## 10. 非目标

以下内容在当前阶段不应被视为产品目标或默认设计范围：

- 品牌故事、品牌人格、品牌主张
- 多用户协作或团队空间
- 企业后台、管理后台、运营后台
- 内容社区、发现页、广场、推荐流
- 商业化套餐、价格体系、订阅策略
- 合规承诺页、行业方案页、销售导向 landing page
- 未被确认的高级编辑能力、批处理能力、项目管理能力

## 12. 对后续 DESIGN.md / UI 的直接约束

后续 `DESIGN.md` 与 UI 设计必须遵守以下约束：

1. 首页默认按“未登录入口 + 登录后工作台”定义，不得先行扩展为宣传首页。
2. 注册页只围绕手机号注册设计，不得加入未经确认的营销区、客群分层区、套餐引导区。
3. 设置页只按账号信息展示与密码管理设计，不得擅自扩展为复杂账户中心。
4. 工作台主视觉与信息层级必须围绕“个人建声 + active voice 管理 + TTS + 历史语音”展开。
5. 下载能力是当前已验证能力，UI 不应把它当作未来占位能力处理。
6. 不得出现多用户、团队、组织、审批、协作相关 UI 暗示。
7. 不得出现价格、套餐、权益、会员、企业版等商业化 UI 暗示。
8. 页面文案必须使用专业、直接、低情绪、面向任务完成的表达。
9. 若设计需要引入任何新模块、新页面、新任务流，必须先补充事实来源，再更新本文档。
