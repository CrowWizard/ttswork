# 运营后台第一版实现计划

## 1. 现状审查

### 1.1 Prisma 现有业务表
当前仓库根目录与 `api-server/` 各维护一份 Prisma schema，结构基本同步，且数据库使用 `relationMode = "prisma"`，不依赖数据库外键。

现有与本次需求直接相关的表：

- `User`
  - 关键字段：`id`、`phoneNumber`、`createdAt`、`freeTtsUsedAt`、`activePureVoiceEnrollmentId`、`activeSceneVoiceEnrollmentId`
- `AnonymousUser`
  - 关键字段：`id`、`lastSeenAt`、`freeTtsUsedAt`、`createdAt`、`activePureVoiceEnrollmentId`、`activeSceneVoiceEnrollmentId`
- `VoiceEnrollment`
  - 对应“声纹/建声记录”
  - 关键字段：`userId`、`anonymousUserId`、`profileKind`、`status`、`voiceId`、`isInvalidated`、`createdAt`
- `TtsJob`
  - 对应“语音生成记录”
  - 关键字段：`userId`、`anonymousUserId`、`voiceEnrollmentId`、`accessKind`、`usageCodeId`、`usageCodeValue`、`status`、`createdAt`
- `UsageCode`
  - 已经具备邀请码/使用码库存与消费追踪语义
  - 关键字段：`code`、`consumedAt`、`consumedByUserId`、`consumedTtsJobId`、`createdAt`
- `Session`
  - 当前已被用于登录会话，不能直接复用为 analytics session

结论：
- “邀请码管理”应优先复用 `UsageCode`，不要重复造 `invitation_codes` / `invitation_code_usages`
- “语音生成记录”应优先复用 `TtsJob`
- “是否生成过声纹”应优先复用 `VoiceEnrollment`
- analytics 需要独立新增轻量模型，避免污染现有登录 `Session`

### 1.2 api-server 目录结构
当前 `api-server` 结构：

- `src/index.ts`
  - Hono 入口，统一挂载路由、请求日志、全局错误处理
- `src/routes/*.ts`
  - 现有业务路由按资源拆分
- `src/lib/*.ts`
  - 共享能力：鉴权、Prisma、配置、日志、错误响应、校验等
- `src/scripts/*.ts`
  - 运营脚本（当前已有批量生成 `UsageCode` 脚本）

当前风格特点：
- 路由文件导出 `createXxxRoutes(cfg)`
- 错误响应统一走 `errorResponse`
- Zod 主要集中在 `src/lib/validation.ts`
- 现有业务逻辑很多直接写在 route 文件里，但本次可新增薄 service 层，避免复杂聚合查询散落在路由内

### 1.3 当前已有相关 API
当前已存在：

- `/api/auth/*`
  - 注册、短信登录、密码登录、获取当前用户、设密、改密、登出
- `/api/voice/profile`
  - 当前用户/匿名用户的 active voice、录音、最近建声
- `/api/voice/recordings`
  - 上传、删除录音
- `/api/voice/enrollments`
  - 创建建声
- `/api/voice/enrollments/recordings/:recordingId/audio`
  - 录音回放
- `/api/voice/enrollments/:enrollmentId/audio`
  - 建声回放
- `/api/voice/enrollments/:enrollmentId/invalidate`
  - 作废声纹
- `/api/tts/scenes`
  - 场景列表
- `/api/tts/usage`
  - 当前权益状态
- `/api/tts`
  - 历史列表、创建语音生成任务
- `/api/tts/:jobId/download`
  - 下载生成语音

结论：
- 当前 API 全部面向“当前登录用户或匿名会话”，缺少跨用户后台查询视角
- 后台接口必须新增 `/api/admin/*`，不能直接在现有 owner-scoped 接口上硬扩展

### 1.4 是否已有后台前端基础设施
仓库前端目前只有：
- `app/page.tsx`
- `app/register/page.tsx`
- `app/settings/page.tsx`

没有发现：
- `app/admin/*`
- 后台 layout / sidebar / table / filter / pagination 体系
- 管理员身份字段或后台鉴权中间件

结论：
- 当前仓库不存在成熟后台前端基础设施
- 但用户已明确要求开始实现 admin 前端页面，因此本轮将补齐“最小可用后台前端”
- admin 前端将独立放在 `app/admin/*` 与 `components/admin/*`，视觉语言不跟随当前工作台
- 仍然避免引入重型 admin 框架，继续使用当前 Next.js + React + Tailwind 基础设施完成

### 1.5 operations-log 规范
仓库根目录已有 `operations-log.md`，采用时间顺序的中文变更记录，内容包含：
- 做了什么
- 为什么这么做
- 验证结果
- 已知限制

结论：
- 本次实现完成后，必须按既有风格追加中文日志记录

  - `admin-analytics-service.ts`

- 返回统一 JSON

- 处理 session 切分、渠道判定、聚合查询、分页查询、批量生成等逻辑

repository 职责：
- 收口较复杂的 Prisma 查询

### 2.4 analytics 落表方案
`POST /api/analytics/collect` 计划按如下流程实现：

1. 使用 Zod 校验：
   - `anonymous_id`
   - `session_id?`
   - `event_name`
   - `url`
   - `referrer?`
   - `utm_source?`
   - `utm_medium?`
   - `utm_campaign?`
   - `user_id?`
   - `timestamp?`
2. 渠道判定：优先 `utm_medium`，其次 `utm_source`，再看 `referrer` 域名，最后 direct/unknown
3. Visitor upsert：
   - 首次写入 first_* 字段
   - 每次更新 `lastSeenAt`
   - 有 `user_id` 时回填 `userId`
4. Session 维护：
   - 按 `anonymousId` 查最近 session
   - 若距离上次 `endedAt` 超过 30 分钟，则新建
   - 若来源字段变化（`utm_source` / `utm_medium` / `utm_campaign` / `referrer` 的归因结果变化），则新建
   - 否则更新最近 session 的 `endedAt`
   - 如请求带 `session_id`，存入 `clientSessionId` 便于前端对齐，但服务端仍以 30 分钟与来源变化规则为准
5. 写入 event：
   - 保存事件、渠道、来源、时间、session 归属

说明：
- 不做复杂事件总线或通用埋点平台抽象
- analytics 只服务于本次运营后台查询目标

### 2.5 业务口径与复用策略
为满足首版可交付且避免重复造实体，计划采用以下口径：

- PV：`AnalyticsEvent.eventName = PAGE_VIEW` 数量
- UV：`AnalyticsVisitor` 去重数量
- Sessions：`AnalyticsSession` 数量
- 新用户数：`User.createdAt` 在时间范围内的数量
- 生成过声纹的用户数：`VoiceEnrollment` 中 `status = READY` 且 `userId` 非空的去重用户数
- 语音生成总次数：`TtsJob` 数量（支持按状态筛选或默认全量）
- 语音生成用户数：`TtsJob.userId` 去重数量
- 使用邀请码的用户数：`UsageCode.consumedByUserId` 去重数量
- 用户查询：只针对注册用户 `User` 做主列表；但支持通过 `anonymous_id` 借助 `AnalyticsVisitor.anonymousId` 反查绑定用户
- 邀请码管理：直接复用 `UsageCode`
- 语音生成记录：直接复用 `TtsJob`

### 2.6 后台鉴权建议
当前项目没有管理员角色模型。首版建议：

- 为 `/api/admin/*` 增加最小 Basic Auth 中间件
- 凭证来源：`ADMIN_USERNAME`、`ADMIN_PASSWORD`
- 如环境变量未配置，管理接口返回明确错误，避免裸开放

原因：
- 不引入新角色表，不改现有用户体系
- 比“完全无鉴权后台”更安全
- 比“临时新增管理员账号体系”更轻量，符合第一版目标

## 4. admin 前端专项计划

### 4.1 目标与范围
本轮 admin 前端只覆盖 4 个页面：

- 概览页：概览卡片 + 趋势图占位式列表化呈现 + 渠道统计表
- 用户页：搜索、分页列表、详情侧栏或详情区块
- 邀请码页：状态筛选、分页列表、批量生成
- 语音生成页：筛选、分页列表、详情区块

不在本轮实现：

- 复杂图表库
- 可拖拽仪表盘
- 多级权限系统 UI
- 可编辑列、导出 Excel、批量操作中心

### 4.2 信息架构
建议采用独立 admin 应用壳：

- 左侧固定导航
  - Overview
  - Users
  - Invite Codes
  - Voice Generations
- 顶部工具条
  - 页面标题
  - 时间范围快捷筛选（仅概览页）
  - Basic Auth 当前连接提示（不展示明文账号）
- 主内容区
  - 概览页：指标 -> 趋势 -> 渠道
  - 列表页：筛选条 -> 数据表 -> 分页 -> 详情

推荐详情交互：

- 用户页与语音生成页优先使用“行内展开详情”或“右侧详情面板”
- 不优先使用 modal，避免频繁打断筛选与翻页

### 4.3 视觉与交互方案
admin 属于 product register，不跟随现有“安静的任务台”浅暖色风格，计划单独采用更明确的运营控制台语言。

方案 A，推荐：深石墨浅文字 + 电蓝 / 青绿强调
- 场景：运营或开发在桌面宽屏上长时间查询、筛选、比对数据
- 优点：和当前前台产品视觉彻底分离，信息密度更高，表格与状态更利于扫描
- 缺点：需要额外处理深色表格对比度与空态层次

方案 B：冷灰白底 + 靛蓝强调
- 场景：白天办公环境下的常规后台
- 优点：实现更稳，和浏览器原生表格/输入控件更接近
- 缺点：容易落入常见 SaaS 后台模板感

推荐结论：
- 采用方案 A
- 原因：用户已明确允许 admin 目录独立，且当前前台已是浅色低干扰风格；后台若继续浅色会与前台边界不清，也更像“同一个工作台加了几个列表页”而非独立运营后台

### 4.4 组件与样式策略
为保持 admin 独立性，计划采用以下方式：

- 保留 Tailwind 作为底层工具，不引入 shadcn/ui、Ant Design、Mantine 等新框架
- 新增 admin 命名空间 class：
  - `admin-shell`
  - `admin-nav-item`
  - `admin-card`
  - `admin-table`
  - `admin-input`
  - `admin-select`
  - `admin-button-primary`
  - `admin-button-secondary`
- 这些 class 写入 `app/globals.css` 的新 `@layer components` 分组中，但不改现有 `.app-*` 组件行为
- admin 主题 token 直接先用 Tailwind 任意值 / 常规类组合落地，不强制修改 `tailwind.config.ts` 扩展一整套新 token，避免把当前全站主题系统搅乱

  - 声纹信息
  - 邀请码信息
  - 输出信息

### 4.6 数据获取与状态管理
- 不引入 React Query/SWR，首版直接使用 `useEffect + fetch + useState`
- 将通用数据获取逻辑收口到 `lib/admin-api.ts`
- 每个页面维护 4 类状态：
  - `loading`
  - `error`
  - `empty`
  - `ready`
- 筛选条件直接映射到 URL query，保证刷新可复现
- 分页参数统一使用：`page`、`pageSize`

### 4.7 Basic Auth 接入方式
- 前端不做登录页，不保存 token
- 直接依赖浏览器对 `/api/admin/*` 的 Basic Auth 质询
- `fetch` 默认携带 `Authorization` Header 的前提是同源请求已通过浏览器缓存凭证
- 若浏览器未缓存凭证，请求失败后页面展示明确错误提示：当前后台需要先通过 Basic Auth 认证

说明：
- 若后续发现浏览器对静态导出场景下的 Basic Auth 体验不稳定，再追加前端输入用户名密码并手动拼接 Header 的方案
- 首版先保持最小复杂度

### 4.8 验证计划补充
除后端验证外，前端新增最小验证：

1. `npm run typecheck`
2. `npm run lint`
3. `npm run build`
4. 确认 `output: "export"` 下 `app/admin/*` 能正常静态导出
5. 检查开发环境下 `/api/admin/*` 能通过 `next.config.ts` rewrite 转发到 `api-server`

### 4.9 风险与取舍

风险 1：当前 `output: "export"` 不适合依赖服务端运行时渲染后台
- 处理：admin 页面全部走客户端数据拉取

风险 2：无现成图表基础设施
- 处理：首版趋势采用简易可读实现，不引入额外图表依赖

风险 3：Basic Auth 对浏览器 `fetch` 体验可能不一致
- 处理：实现时优先验证同源请求行为，若不稳定再降级为前端手动输入凭证方案

风险 4：admin 与前台共用 `globals.css`
- 处理：通过独立 `admin-*` class 命名空间隔离，不复用 `.app-*` 组件

## 5. 关键风险与处理

### 风险 1：管理员体系当前不存在
处理：
- 使用环境变量驱动的 Basic Auth 作为首版保护层

### 风险 2：`anonymous_id` 不是现有业务主键

- 将其定义为 analytics visitor 主键，不强行映射到 `AnonymousUser.id`

- 根目录与 `api-server/` 同步修改，避免后续 `prisma` 命令读错 schema

### 风险 4：当前无数据库外键
处理：
- 所有后台联查都按应用层字段关联编排，不依赖 FK

## 6. 推荐实施结论
推荐按以下边界落地：

- 做完整后端第一版：analytics + admin 查询 + invite code 生成/查询
- 同时补齐最小 admin 前端：概览页、用户页、邀请码页、语音生成页
- 管理接口统一收敛到 `/api/admin/*`
- analytics 只做 visitor / session / event 三张轻量表，不扩展成通用埋点平台
- admin 前端独立于当前前台风格，采用单独的信息密度与视觉语言，但继续复用 Next.js + Tailwind 基础设施
