# 语音工作台权益与 UI 调整实施计划

## 目标范围

1. 首页去掉模块切换 `nav`，不再暴露“B站视频分析”。
2. 交换“音频上传”和“录音”两个按钮的视觉颜色。
3. 删除录音素材时同步删除/作废基于该录音生成的纯粹版与场景版声纹；前端不再提供单独删除/作废声纹按钮。
4. 文本转语音输入不再要求使用码。
5. 登录用户去掉试用；新注册/短信自动创建用户赠送 100 积分。
6. 登录用户新增积分余额；使用码可兑换积分，每个使用码兑换 200 积分；header 支持输入使用码快速兑换。
7. 每次文本转语音消耗 20 积分。

说明：第 7 条“语音换文本”按当前系统现有能力“文本转语音 / TTS”实现。

## 方案对比

### 方案 A：仅在 `User` 上保存积分余额

优点：改动小，查询简单。
缺点：无法审计每次赠送、兑换、消费来源；发生并发扣费或运营核账时缺少依据。

### 方案 B：`User.pointsBalance` + `PointTransaction` 流水（推荐）

优点：余额查询快，流水可审计注册送积分、使用码兑换、TTS 消费；便于后台扩展和排查并发问题。
缺点：需要新增一张表和更多事务写入。

采用方案 B。

## 数据模型计划

1. 同步修改两份 Prisma schema：`prisma/schema.prisma` 与 `api-server/prisma/schema.prisma`。
2. `User` 新增 `pointsBalance Int @default(0)`，移除业务对 `freeTtsUsedAt` 的依赖；字段是否物理删除需谨慎：
   - 推荐保留 `freeTtsUsedAt` 字段但不再使用，减少现有数据库破坏性迁移风险。
   - 文档标记为旧试用字段。
3. 新增枚举 `PointTransactionType`：
   - `REGISTER_BONUS`
   - `USAGE_CODE_REDEEM`
   - `TTS_CONSUME`
4. 新增模型 `PointTransaction`：
   - `id`
   - `userId`
   - `type`
   - `delta`
   - `balanceAfter`
   - `usageCodeId?`
   - `ttsJobId?`
   - `createdAt`
   - 索引：`userId + createdAt`、`usageCodeId`、`ttsJobId`
5. `UsageCode` 保持一次性消费语义，兑换时写入 `consumedAt` / `consumedByUserId`；`consumedTtsJobId` 将仅用于旧 TTS 直接消费记录，新兑换流程不再写它。
6. 更新 `api-server/init-db.sql`、`api-server/DATABASE_DICTIONARY.md`、示例配置/README 中与试用、使用码直接生成相关描述。

## 后端接口计划

### 注册与登录

1. `api-server/src/routes/auth.ts`：
   - `/auth/register` 创建新用户时设置 `pointsBalance: 100`，同时写 `PointTransaction(REGISTER_BONUS, +100)`。
   - `/auth/login/sms` 当前使用 `upsert`，需要区分“新创建用户”和“已有用户”。推荐改为事务：先查手机号，不存在则 create 并赠送 100 积分，存在则只更新 `phoneVerifiedAt`。
   - 密码登录不赠送积分。
2. `buildAuthUser` 返回 `pointsBalance`，前端 header 可直接显示。
3. 匿名数据迁移不迁移免费试用状态；去掉 `migrateAnonymousDataToUser` 中 `freeTtsUsedAt` 迁移逻辑。

### 积分与使用码兑换

1. 新增用户权益接口，建议挂在现有 TTS 路由下或新增 `credits` 路由：
   - `GET /api/tts/usage` 改为返回 `{ isAuthenticated, pointsBalance, ttsCostPoints: 20, usageCodeRedeemPoints: 200 }`，不再返回免费次数/试用限制。
   - 新增 `POST /api/tts/usage/redeem` 或 `POST /api/credits/redeem`，body `{ usageCode }`。
2. 兑换事务逻辑：
   - 必须登录，匿名用户返回 401。
   - 校验 6 位使用码格式。
   - 查询 `UsageCode` 未消费且 module 为 `VOICE_TO_TEXT`。
   - 原子更新 `UsageCode.consumedAt/consumedByUserId`。
   - `User.pointsBalance += 200`。
   - 写 `PointTransaction(USAGE_CODE_REDEEM, +200, balanceAfter, usageCodeId)`。
   - 返回最新余额。
3. 保留通用使用码配置 `USAGE_CODE_GENERAL_CODE` 的处理需改造：
   - 推荐移除“通用码无限生成”行为，避免绕过积分体系。
   - 如业务仍需通用码，应也通过兑换接口增加积分，但不能写 `UsageCode` 一次性消费记录；该点实施时按当前需求优先移除直接生成逻辑。

### TTS 消费

1. `api-server/src/routes/tts.ts`：
   - 移除匿名/登录免费试用分支、`FREE_TTS_TEXT_LIMIT`、使用码直接生成分支、`GENERAL_USAGE_CODE` 生成分支。
   - 要求登录用户才能生成；若匿名提交，返回“请先登录后使用积分生成”。
   - 文本上限统一保留现有 schema 500 字。
   - 在 Serializable 事务中锁定/读取最新用户余额，余额不足 20 则拒绝。
   - 创建 `TtsJob` 时 `accessKind` 可继续使用旧枚举值中的 `USAGE_CODE` 不合适；推荐新增 `POINTS` 到 `TtsAccessKind`，并将新任务标记为 `POINTS`。
   - 扣减 `User.pointsBalance -= 20`，写 `PointTransaction(TTS_CONSUME, -20, balanceAfter, ttsJobId)`。
   - Qwen 合成失败后的积分处理需要明确：推荐“失败退回积分”，即把 job 标为 `FAILED` 后再事务返还 20 分并写一条反向流水；或在外部调用成功后再扣分。为避免生成成功但扣分失败，推荐先扣费，失败自动退回。
2. `GET /api/tts` 历史返回的来源文案从免费/使用码调整为积分。
3. `api-server/src/lib/validation.ts`：`ttsRequestSchema` 移除 `usageCode` 字段；新增兑换请求 schema。

## 风险与回滚

1. 数据迁移风险：新增字段/表为非破坏性；旧 `freeTtsUsedAt` 推荐保留，方便回滚。
2. 并发风险：兑换与扣费必须在事务中做条件更新，防止同一使用码重复兑换或余额并发超扣。
3. 外部 TTS 失败：扣费后若 Qwen 失败必须退回积分并记录流水，避免用户资产损失。
4. 回滚方式：保留旧字段和旧使用码表，若需要回滚可恢复前端 usage code 输入与后端旧分支；新增积分表不影响旧流程。
