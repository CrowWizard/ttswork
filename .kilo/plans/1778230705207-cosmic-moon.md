# 视频分析结构块增强与预计完成时间 Plan

## 背景结论

当前 `/video-analysis` 链路中，结构块来自 `video-analysis-worker/services/analyzer.py` 的 `StructureExtract.structural_blocks`，目前 schema 仅支持 `hook/promise/re_hook/cta` 字符串时间范围与 `meat` 字符串数组。API 在 `api-server/src/routes/video-analysis.ts` 中透传 `scriptAnalysis` / `metadataJson`，前端 `components/bilibili-analysis/bilibili-analysis-workspace.tsx` 只展示“LLM 返回位置”。

因此本次不能只改前端，需要从 LLM 输出结构、worker 校验与后处理、API DTO、前端展示一起改。

## 目标

1. 结构块由“位置字符串”升级为“可执行诊断块”：包含分数、时间范围、百分比、摘要、优点、不足、具体修改建议。
2. 前端能直接告诉创作者结构块出现在视频第几秒或百分之多少，并能展示建议应落在第几秒或百分比。
3. 根据抓取到的视频信息和任务状态，给用户一个大约等待时间提示，例如“预计还需要约 3 分钟，可以先去忙其他，稍后回来查看”。
4. 保留历史任务兼容能力：旧任务中已持久化的字符串结构块仍能展示，不强制迁移历史 JSON。

## 推荐方案

采用“小步增强，不加新数据库字段”的方案：

- 结构块详情直接进入现有 `scriptAnalysis.structural_blocks` 和 `metadataJson.structural_blocks` JSON。
- 预计等待时间由 API 基于 `VideoSource.durationSeconds`、字幕/转写状态、当前阶段、阶段事件耗时实时计算后返回给前端。
- 不新增 Prisma 字段，避免数据库迁移风险；如果后续要做统计，再把估算字段持久化。

备选方案是新增 `estimatedSeconds`、`estimatedReadyAt` 等数据库字段，由 worker 在 `SNAPSHOT_FETCH` 后写入。优点是历史可追踪，缺点是需要 schema 迁移和更多写入路径。本需求更适合先用 API 派生估算。

## 数据结构设计

### LLM 结构块输出

将 `StructuralBlocksOutput` 从字符串升级为对象：

```json
{
  "structural_blocks": {
    "hook": {
      "name": "开头抓注意力",
      "time_range": { "start": "00:00", "end": "00:08" },
      "score": 82,
      "summary": "用痛点问题快速建立观看动机。",
      "strengths": ["开场问题明确", "与目标用户痛点相关"],
      "weaknesses": ["利益承诺出现偏晚"],
      "suggestions": [
        {
          "type": "script",
          "target_time": "00:05",
          "content": "建议补一句看完收益，让观众知道继续看的理由。"
        }
      ]
    },
    "promise": null,
    "meat": [
      {
        "name": "主体内容 1",
        "time_range": { "start": "00:15", "end": "01:20" },
        "score": 76,
        "summary": "解释主要观点，但中段信息密度偏高。",
        "strengths": ["步骤完整"],
        "weaknesses": ["连续概念较多"],
        "suggestions": [
          {
            "type": "pacing",
            "target_time": "00:48",
            "content": "这里建议插入一个例子或小结，降低理解负担。"
          }
        ]
      }
    ],
    "re_hook": null,
    "cta": null
  }
}
```

### 后端后处理补充字段

LLM 负责返回时间、评分、优点、不足和建议；worker 负责基于视频时长计算更稳定的秒数和百分比：

- `start_seconds`
- `end_seconds`
- `start_percent`
- `end_percent`
- `suggestions[].target_seconds`
- `suggestions[].target_percent`

这样前端不凭空推断内容，只展示后端已经规范化后的结果。

### 预计等待时间 DTO

API 在 `VideoAnalysisJobDetail` 中增加派生字段：

```ts
estimate: {
  totalSeconds: number | null;
  remainingSeconds: number | null;
  readyAt: string | null;
  confidence: "low" | "medium" | "high";
  message: string;
} | null
```

估算依据：

- `source.durationSeconds`：抓取视频信息后可用，视频越长预计越久。
- `source.transcriptStatus` / `source.subtitleStatus` / `source.transcriptSource`：已有字幕比 ASR 快，ASR 或字幕不可用时更久。
- `stageEvents` 已完成阶段耗时：用于扣减已消耗时间。
- `currentStage`：越靠后，置信度越高。

## 实施步骤

### 1. Worker schema 与 prompt

修改 `video-analysis-worker/services/analyzer.py`：

- 新增 `StructureBlockTimeRange`、`StructureBlockSuggestion`、`StructureBlockDetail` Pydantic model。
- 修改 `StructuralBlocksOutput`：`hook/promise/re_hook/cta` 支持 `StructureBlockDetail | None`，`meat` 改为 `list[StructureBlockDetail]`。
- 更新 `STEP1_EXAMPLE`，让 LLM 明确返回结构块评分、摘要、优点、不足和建议。
- 更新 `_step1_extract_structure` prompt 约束：
  - `score` 为 0 到 100。
  - `strengths`、`weaknesses`、`suggestions` 每项建议 1 到 3 条。
  - `suggestions.type` 限定为 `script/editing/subtitle/pacing/cta`。
  - 所有结构块必须尽量引用字幕对应时间。
- 更新 `_build_mock_report`，让 mock 模式也生成新结构块。

### 2. Worker 归一化与百分比计算

在 `analyzer.py` 增加结构块后处理：

- `_normalize_structure_payload` 中兼容旧式字符串：如果 LLM 偶尔返回 `"00:00-00:05"`，转换成最小对象，避免一次格式偏差直接失败。
- `_postprocess_metadata` 或新增 `_postprocess_structure_blocks`：
  - 解析 `time_range.start/end` 为秒数。
  - 根据 `total_duration` 计算百分比，边界控制在 0 到 100。
  - 为建议里的 `target_time` 计算 `target_seconds` 和 `target_percent`。
  - 修正 `end_seconds < start_seconds` 的异常范围。
- 日志中记录结构块数量、低分结构块数量、归一化 warning 数量，方便排查 LLM 输出质量。

### 3. 测试覆盖

更新 `video-analysis-worker/tests/test_analyzer_schema.py`：

- 新结构块完整 JSON 能通过 `StructureExtract` 校验。
- 缺少必填字段仍稳定抛 `AnalysisResultFormatError`。
- 旧字符串结构块可被归一化为对象。
- 百分比计算边界正确，例如 30 秒视频中 `00:03` 为 10%。

必要时补充 `test_transcript_strategy.py` 中 mock 输出断言，确保主流程仍能完成。

### 4. API 派生预计等待时间

修改 `api-server/src/routes/video-analysis.ts`：

- `toVideoAnalysisJobDetailDto` 增加 `estimate` 字段。
- `loadStageEventsByJobId` 当前在详情接口已可用，估算函数应接收 `job/source/stageEvents`。
- 新增 `buildVideoAnalysisEstimate`：
  - READY/FAILED 返回 `remainingSeconds: 0` 或 `null`。
  - 未抓到 `durationSeconds` 时返回低置信度通用提示。
  - 有时长后按阶段估算：
    - 元信息/字幕阶段：基础 60 到 120 秒。
    - 有字幕：按视频时长低倍率估算。
    - ASR 或字幕不可用：按视频时长高倍率估算。
    - LLM 阶段：按字幕长度不可得时用视频时长兜底。
  - 根据已完成 `stageEvents.durationMs` 扣减剩余时间。
  - 输出面向用户的中文 `message`。
- 列表接口可暂不加 estimate，避免最近任务列表变重；详情页轮询已有足够信息。

### 5. 前端类型与展示

修改 `components/bilibili-analysis/bilibili-analysis-workspace.tsx`：

- 更新 `ScriptAnalysis.structural_blocks` 与 `MetadataJson.structural_blocks` 类型，支持新对象和旧字符串两种形态。
- 新增前端展示类型转换函数：
  - `normalizeStructureBlockRows` 只负责把后端对象转换成表格行，不再生成评价内容。
  - 旧字符串任务仅展示“LLM 返回位置”，新对象展示完整诊断。
- 改造 `ScriptBlocksTable`：
  - 桌面表格列建议为：结构块、出现位置、评分、优点、不足、具体改法。
  - 移动端继续使用 `MobileStackedRows`，字段包含秒数、百分比、评分、优点、不足、建议。
  - 评分可用文本等级辅助：`>=85 表现好`、`70-84 可优化`、`<70 优先修改`。
- 可选增强：在结构块表格上方增加轻量时间轴，按 `start_percent/end_percent` 展示结构块位置；仅在新对象有百分比时显示。

### 6. 前端预计等待时间提示

修改前端任务状态区域：

- `VideoAnalysisJobDetail` 增加 `estimate` 类型。
- `AnalysisPendingState` 中在“当前阶段”旁显示估算提示：
  - 有 `remainingSeconds`：`预计还需要约 X 分钟，可以先去忙其他，稍后回来查看。`
  - 低置信度：`正在获取视频信息，拿到时长后会给出更准确的预计时间。`
- `StageProgressPanel` 中补充估算信息，避免用户只看到阶段列表但不知道是否要等很久。
- 保留 `aria-live="polite"`，让状态变化对辅助技术可感知。

### 7. 文档与操作记录

实施时需要更新：

- `operations-log.md`：记录结构块 schema、LLM prompt、API estimate、前端展示与验证结果。
- 如发现 `api-server/prisma/schema.prisma` 与根目录 `prisma/schema.prisma` 仍存在视频分析模型差异，需要单独记录风险；本计划不主动合并根 schema，除非实施中确认根 schema 仍被当前构建/部署使用。

## 验证计划

本地验证建议：

1. `python -m unittest video-analysis-worker/tests/test_analyzer_schema.py`
2. `python -m unittest video-analysis-worker/tests/test_transcript_strategy.py`
3. `npm run lint`
4. `npm run typecheck`
5. `npm run build`
6. `cd api-server && bun run build`

如果涉及 Prisma client 类型变化才执行：

1. `cd api-server && bunx prisma generate`

本计划推荐不改数据库 schema，因此不需要 `prisma:push`。

## 风险与处理

- LLM 新结构更复杂，格式失败概率会上升：通过更明确示例、schema 归一化和单测降低风险。
- 历史任务结构块是字符串：前端和 worker 归一化都需要兼容旧格式。
- 预计等待时间只能是估算：必须展示置信度语义，避免承诺精确完成时间。
- ASR 耗时波动大：估算函数应保守，宁可提示“约 5 分钟”而不是过度精确到秒。

## 交付结果

完成后，创作者会看到：

- 每个结构块在视频中的秒数与百分比位置。
- 每个结构块的评分、优点、不足和可执行修改建议。
- 建议具体应该改在第几秒或视频百分之多少位置。
- 分析进行中时，系统根据视频信息提示大概还要等多久，可以先去处理其他事情。
