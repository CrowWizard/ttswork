# 前端网站审计计划

## 目标
对当前 Next.js 网站做一次基于 `/impeccable audit` 方法论的技术型前端审计，输出可执行的改进建议，覆盖可访问性、性能、响应式、主题系统与设计反模式五个维度。

## 当前已确认上下文
- 项目类型：Next.js 15 + React 19 + Tailwind CSS 的单页语音工具站点。
- 主要界面入口：`app/page.tsx`，核心实现位于 `components/voice-studio.tsx`。
- 相关页面：`app/register/page.tsx`、`app/settings/page.tsx`。
- 全局样式与主题来源：`app/globals.css`、`tailwind.config.ts`。
- 根据界面形态与任务属性，当前任务注册类型推断为 `product`：这是一个登录后执行任务的工具型界面，设计服务于操作流程。

## 已识别风险与前置缺口
1. `PRODUCT.md` 缺失，无法按 `/impeccable` 要求加载完整品牌、用户、语气与战略原则。
2. `DESIGN.md` 缺失，无法获得现有颜色、字体、组件与层级规范。
3. 仓库中不存在 `.opencode/skills/impeccable/scripts/load-context.mjs`，说明项目未按技能约定完成本地上下文接入。
4. 因处于计划模式，只能记录该缺口，不能执行 `/impeccable teach` 或 `/impeccable document`。

## 审计范围
### 页面范围
- 首页工作台：登录、建声录音、TTS、历史记录
- 注册页：手机号、验证码、可选密码
- 设置页：账户信息、密码设置/修改

### 代码范围
- `components/voice-studio.tsx`
- `app/register/page.tsx`
- `app/settings/page.tsx`
- `app/layout.tsx`
- `app/globals.css`
- `tailwind.config.ts`

## 审计方法
### 方案 A：静态代码审计（推荐）
基于现有源码定位问题，形成带文件与行号的审计报告。

优点：
- 当前计划模式下可完整执行
- 可直接给出精确证据位置
- 对可访问性、响应式、主题一致性、反模式识别足够有效

局限：
- 无法得到真实运行时对比度、交互延迟、视觉回归截图
- 某些移动端问题只能做代码级推断

### 方案 B：运行态审计
启动应用后再结合浏览器进行视口、交互、可访问性和感知质量检查。

优点：
- 可验证真实 UI 呈现与交互链路
- 更容易发现滚动、溢出、触控目标、状态切换问题

局限：
- 当前计划模式不适合启动长驻进程与执行运行态检查
- 依赖本地服务与数据准备

### 结论
本轮计划采用方案 A，后续执行阶段如需要更高置信度，可补充方案 B 做二次复核。

## 具体执行步骤
1. 补齐上下文判断
- 以 README 与现有页面职责替代 `PRODUCT.md`，临时推断产品定位与用户任务。
- 在报告中明确说明品牌上下文缺失会降低“是否足够 on-brand”的判断精度。
- 按技能要求给出一次性提示：建议后续运行 `/impeccable teach` 与 `/impeccable document`。

2. 完成五维审计评分
- Accessibility：检查语义结构、表单标签、焦点可见性、键盘可达性、错误反馈、ARIA 与状态表达。
- Performance：检查渲染热点、定时器频率、客户端逻辑体量、无谓 re-render、重客户端页面与可拆分机会。
- Theming：检查硬编码颜色、字体、阴影、`color-scheme`、暗色主题缺失、token 复用程度。
- Responsive Design：检查固定宽度、最小触控目标、窄屏布局折叠、长文本截断、音频控件与操作按钮在移动端的可用性。
- Anti-Patterns：按 `/impeccable` 规则检查 AI 感、玻璃态默认使用、重复卡片结构、过度依赖圆角白卡、弱层级与模板化布局。

3. 输出审计报告结构
- Audit Health Score 总表（5 维 20 分）
- Anti-Patterns Verdict（先给是否有明显 AI 感结论）
- Executive Summary（总分、问题数量、最高优先级问题、改进方向）
- Detailed Findings by Severity（P0-P3，逐条给出位置、影响、标准、建议与建议命令）
- Patterns & Systemic Issues（系统性问题）
- Positive Findings（保留项）
- Recommended Actions（按优先级列出 `/impeccable` 命令）

## 初步观察，后续会在正式审计中落为证据
1. 主题系统较弱：`tailwind.config.ts` 仅定义了少量十六进制颜色，`app/globals.css` 固定 `color-scheme: light`，说明主题扩展能力不足。
2. 组件风格高度重复：登录、注册、设置、工作台均复用“浅色背景 + 白卡 + 大圆角 + 柔阴影”模板，容易出现工具站模板化观感。
3. 首页单组件过大：`components/voice-studio.tsx` 体量很大，状态和流程高度耦合，后续既影响性能判断，也影响可维护性与一致性。
4. 交互可达性可能存在缺口：录音主按钮主要依赖 `onMouseDown/onMouseUp/onTouchStart/onTouchEnd`，需要重点检查纯键盘操作路径。
5. 加载与空状态表达偏弱：多个区域直接显示“加载中...”或空文案，缺少更明确的 skeleton、引导和状态分层。

## 预期交付结果
- 一份可直接执行的中文审计结论
- 每个关键问题都带代码定位与影响说明
- 一组按优先级排序的改进建议，而不是泛泛而谈的设计意见
- 在品牌上下文缺失的前提下，尽量把建议收敛到可验证的技术与交互质量问题

## 执行完成后的建议顺序
1. `/impeccable teach`：补齐 `PRODUCT.md`
2. `/impeccable document`：生成 `DESIGN.md`
3. `/impeccable audit`：基于现有代码输出正式审计结果
4. 按审计结论再执行 `/impeccable adapt`、`/impeccable harden`、`/impeccable layout`、`/impeccable clarify`、`/impeccable polish`
