# 首页认证与个人设置改造计划

## 目标

1. 缩短首页打开时的登录状态检查体感延迟。
2. 首页同时支持短信登录与用户名密码登录。
3. 首页提供“注册”链接，跳转到独立注册页面。
4. 增加“个人设置”页面，支持设置密码与修改密码；修改密码前必须完成短信验证码校验。

## 当前现状与证据

1. 首页当前只有 `app/page.tsx -> components/voice-studio.tsx` 一个入口，未登录态仅渲染短信登录表单。
2. 前端在 `VoiceStudio` 挂载后通过 `fetch("/api/auth/me")` 恢复登录态，`authLoading` 为 `true` 时整页只显示“正在检查登录状态...”。因此用户必须等待浏览器 hydration 和一次接口往返后才能看到真实页面。
3. `next.config.ts` 当前启用了 `output: "export"`，这意味着前端默认走静态导出，无法像 SSR/动态渲染那样在服务端首屏直接判定登录态。
4. 后端已具备以下能力：
   - `POST /api/auth/register`
   - `POST /api/auth/login/password`
   - `POST /api/auth/login/sms`
   - `POST /api/auth/sms/send`
   - `GET /api/auth/me`
   - `POST /api/auth/logout`
5. Prisma 的 `User` 模型已有 `passwordHash`、`phoneNumber`、`phoneVerifiedAt`，适合继续扩展密码设置/修改能力。
6. 现有短信场景只有 `register` 与 `login`，修改密码还缺少独立短信场景与对应校验接口。

## 推荐方案

### 方案 A（推荐，兼容当前静态导出）

保留 `output: "export"`，通过前端状态分层降低“慢”的感知，而不是强依赖首屏拿到登录态。

做法：
1. 首页首屏立即展示通用工作台外壳与登录表单骨架，不再整页阻塞在 `authLoading`。
2. 将登录态恢复改为“后台检查 + 局部切换”，避免用户先看到整屏“正在检查登录状态...”。
3. 若已登录，在接口返回后无闪烁切换到工作台；若未登录，则继续停留在登录/注册入口。

优点：
1. 不破坏现有静态导出部署。
2. 改动范围小，风险低。
3. 与当前 Nginx/静态发布方式兼容。

缺点：
1. 无法做到真正意义上的服务端首屏鉴权。
2. 已登录用户仍需要一次 `/api/auth/me` 请求，只是体感更快。

### 方案 B（更彻底，但需调整部署）

取消 `output: "export"`，改为 Next.js 动态渲染或 SSR，在服务端读取 Cookie 判定登录态。

优点：
1. 已登录用户首屏即可拿到真实状态。
2. 认证页与工作台切换更自然。

缺点：
1. 会改变当前前端部署模型，不再是纯静态导出。
2. 需要同步调整发布、代理与运维脚本，影响面更大。

### 如果改成 SSR，需要调整哪些部署链路

1. 取消前端静态导出后，`make build-frontend` 与当前“发布到 `/var/www/voice-mvp` 静态目录”的流程需要改成 Node/Next 运行时部署。
2. 需要在服务器长期运行 Next.js 进程，例如 `next start` 或独立 Node 服务，并由 systemd 托管。
3. Nginx 不再只托管静态文件，而是需要把前端请求反向代理到 Next.js 服务端口。
4. 当前开发态依赖 rewrite 把 `/api/*` 转到 `:3001`；生产若采用 SSR，需要明确：
   - 前端页面请求走 Next.js 服务
   - API 请求继续走 Hono/Bun 服务，或由 Next.js 再转发
5. 自动更新脚本也要同步调整，不再只是覆盖静态产物，还要：
   - 安装前端运行时依赖
   - 构建 `.next`
   - 重启 Next.js 服务
6. 运维复杂度会明显高于当前静态方案，因此不建议仅为“首页登录态恢复偏慢”就立刻切到 SSR。

## 实施计划

### 一、前端认证流重构

1. 将 `VoiceStudio` 中“仅短信登录”的认证区拆成可切换的双模式表单：
   - 短信登录
   - 用户名密码登录
2. 新增前端认证状态字段，至少区分：
   - `authResolving`：后台恢复中
   - `authMode`：`sms` / `password`
   - `authForm`：手机号、验证码、密码
3. 调整首页加载体验：
   - 删除当前整页阻塞式 `authLoading` 页面
   - 未知登录态时先渲染首屏框架与认证卡片占位
   - `/api/auth/me` 返回后再细化为登录页或工作台
4. 保持已登录后的现有建声、TTS、退出逻辑不回归。

### 二、注册页面落地

1. 新增独立注册路由页面，例如 `app/register/page.tsx`。
2. 注册表单包含：
   - 手机号
   - 短信验证码
   - 密码（已确认：可选）
3. 注册页调用既有后端能力：
   - 发送验证码：`scene = register`
   - 提交注册：`POST /api/auth/register`
4. 注册成功后沿用后端自动建会话逻辑，直接跳回首页并进入已登录工作台。
5. 首页登录卡片增加“注册”链接，跳转到注册页。
6. 若用户注册时未填写密码，后续首页密码登录要继续保留“该账号尚未设置密码，请使用短信登录”的现有后端提示，并在个人设置页补齐设置密码入口。

## 三、个人设置页面

1. 新增独立设置页，例如 `app/settings/page.tsx`。
2. 页面进入时先校验登录态，未登录则跳回首页。
3. 页面展示当前账号基础信息：
   - 手机号
   - 是否已设置密码
4. 根据账号状态分两块能力：
   - 未设置密码：展示“设置密码”表单
   - 已设置密码：展示“修改密码”表单
5. 首页右上角当前账号区域增加“个人设置”入口。

### 四、后端密码管理能力补齐

1. 在 `api-server/src/lib/validation.ts` 新增 schema：
   - 设置密码请求
   - 修改密码请求
   - 修改密码短信校验请求
2. 在后端新增密码管理接口，建议收敛在 `/api/auth` 下：
   - `POST /api/auth/password/set`
   - `POST /api/auth/password/change`
3. 规则设计：
   - 设置密码：仅允许已登录且当前 `passwordHash` 为空的用户操作
   - 修改密码：仅允许已登录用户操作，且必须先完成短信验证码校验
4. 为避免前端保存一次性“已校验”状态过于脆弱，推荐后端一次请求内完成：
   - 校验登录用户手机号 + 短信码 + 新密码
   - 校验通过后直接更新 `passwordHash`

## 五、短信场景扩展

1. 扩展 Prisma `SmsScene` 枚举，新增密码修改场景，例如：`PASSWORD_CHANGE`。
2. 扩展 `api-server/src/lib/sms.ts`：
   - `SmsSceneValue` 增加 `password_change`
   - 场景枚举映射与短信方案名读取逻辑同步扩展
3. 扩展配置：
   - `SMS_PASSWORD_CHANGE_SCHEME_NAME`
   - `config.example.yaml` / `.env.example` / README 同步说明
4. 扩展发送验证码接口的场景白名单与错误文案。

## 六、数据库与配置影响

1. Prisma schema 需要修改 `SmsScene` 枚举。
2. 需要执行 Prisma generate 与 `db push`。
3. 若生产使用真实短信，还需要补充密码修改场景的短信方案名配置；若仍为 mock 模式，可直接沿用本地调试返回验证码。

## 七、实现顺序

1. 先改后端：补齐短信场景、校验 schema、密码设置/修改接口。
2. 再改前端首页：消除整页阻塞登录态检查，加入双登录模式与注册入口。
3. 新增注册页面并接入注册接口。
4. 新增个人设置页面并接入设置/修改密码接口。
5. 最后补文档与 `operations-log.md`，并做本地验证。

## 验证清单

1. 未登录打开首页：应立即看到首屏内容，不再长时间停留在整页“检查登录状态”。
2. 已登录打开首页：在登录态恢复后自动进入工作台，功能不回归。
3. 短信登录：已注册和未注册手机号行为符合设计。
4. 用户名密码登录：仅对已设置密码账号可用。
5. 注册流程：验证码发送、注册成功、自动登录成功。
6. 设置密码：无密码账号可设置成功。
7. 修改密码：必须短信校验成功后才可修改成功。
8. 修改密码后：旧密码失效，新密码可登录。
9. 退出登录后：设置页受保护，不能直接访问敏感操作。
10. 本地执行：`lint`、`typecheck`、必要时 `build` 与 Prisma 相关命令通过。

## 风险与注意事项

1. 当前项目依赖静态导出部署，若采用 SSR 方案，需要连带调整部署链路，风险明显高于前端体感优化方案。
2. 登录页、注册页、设置页三者应统一认证文案与错误提示，避免同一手机号在不同页面规则不一致。
3. 修改密码场景涉及真实短信模板时，需要确认阿里云控制台已创建对应方案名。
4. 注册密码已确认可选，因此前端和设置页都要覆盖“已注册但未设密码”的分支体验。
5. 实施时需要继续遵守现有不回退其他用户改动、逐步验证、同步记录 `operations-log.md` 的要求。

## 结论

1. 从当前仓库结构与部署方式看，推荐按方案 A 实施：保留静态导出，只优化首页登录态恢复体验。
2. 注册页密码按“可选”处理，但要确保密码登录失败提示与设置密码入口闭环完整。
