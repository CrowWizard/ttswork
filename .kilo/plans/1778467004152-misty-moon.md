# 视频文案生成 MVP 实施计划

## 目标

实现第三个功能：视频文案生成。第一版只支持基于用户输入话题生成 `video_script`，严格按 Mastra Prompt 规范的 `0→7` 工作流产出进度标签、脚本文本和最终 JSON。

本阶段明确不做：

- 图片生成
- 点数消耗
- `social_post`
- 发布时间建议
- 热门 hook 数据库查询

## MVP 范围

第一版只开放视频脚本生成。

输入参数：

```json
{
  "topic": "话题",
  "type": "video_script",
  "platform": "bilibili | youtube | xiaohongshu | douyin | weixin-channels | twitter",
  "language": "zh-CN | en-US",
  "tone": "personal | company | professional-casual",
  "verbosity": "concise | detailed",
  "duration": "short | medium | long",
  "heroOpening": "可选",
  "outroClosing": "可选"
}
```

默认值：

```json
{
  "type": "video_script",
  "platform": "bilibili",
  "language": "zh-CN",
  "tone": "personal",
  "verbosity": "concise",
  "duration": "short"
}
```

第一版不接收：

```json
{
  "format": "social_post 专用，暂不接收",
  "isPicture": "图片生成暂不接收"
}
```

## 最终输出契约

第一版最终只返回 `video_script` Schema，不实现 `social_post` Schema。

```json
{
  "status": "completed",
  "metadata": {
    "topic": "string",
    "platform": "string",
    "language": "zh-CN",
    "estimatedDurationSec": 180,
    "sections": [
      {
        "name": "hero",
        "density": "Impact",
        "durationSec": 20
      }
    ]
  },
  "script": "[SECTION:hero]\n完整脚本...",
  "hook": "string",
  "references": "string",
  "warnings": [],
  "progress_log": [
    {
      "step": "research_done",
      "ts": 1715000000000,
      "message": "已收集 3 条关键信息"
    }
  ]
}
```

说明：

- 不输出 `image_prompts`，因为图片生成暂不做。
- 不输出 `suggested_date`，因为发布时间建议暂不做。
- 不输出 `social_content`，因为 `social_post` 暂不做。
- `script` 必须包含完整 `[SECTION:xxx]` 标记。
- JSON 外部不得包含 Markdown、解释文本或代码块。

## 总体方案

复用现有 `video-analysis` 的异步任务模式，但 Worker 改为 TypeScript 实现：

1. 前端创建内容生成任务。
2. 后端写入 `ContentGenerationJob`。
3. TypeScript Worker 领取任务并按 `0→7` 执行。
4. 每完成一步写入 stage event，并同步维护 `progress_log`。
5. 前端支持轮询任务详情；可选支持 SSE 流式进度。
6. 任务完成后展示最终 JSON 和脚本文案。

## 执行工作流

### 步骤 0：加载用户偏好 `[PREFERENCES]`

任务：合并默认配置、用户历史偏好和本次请求。

第一版没有独立用户偏好表时，使用请求参数 + 默认值。

输出标签：

```text
[PREFERENCES] Platform: {platform} | Language: {language} | Tone: {tone} | Verbosity: {verbosity} | HeroOpening: {heroOpening} | OutroClosing: {outroClosing}
```

### 步骤 1：定义方向与研究话题 `[DIRECTION]` & `[RESEARCH]`

任务：定义受众、风格、范围、语调、时长，并执行网络搜索。

搜索词：

```text
{topic} 最新发展 {current_year}
```

输出标签：

```text
[DIRECTION] 受众-{audience} | 风格-{style} | 范围-{scope} | 语调-{tone} | 时长-{duration}
[RESEARCH] • 研究要点 1 • 研究要点 2 • 研究要点 3
```

规则：

- 优先收集最近一个月信息。
- 至少尝试收集 3 条高质量内容。
- 不足 3 条时继续生成，但写入 warning 和低置信度 metadata。
- 搜索失败时降级为模型通用知识摘要，并写入 warning。

### 步骤 2：分类与主题匹配 `[CATEGORY]` & `[TOPIC]`

任务：匹配预设分类库；第一版无分类库时由 LLM 自主生成简洁分类。

输出标签：

```text
[CATEGORY] {category}
[TOPIC] {topic}
```

第一版不新增分类库表，只把结果保存到任务表。

### 步骤 3：获取风格参考 `[POPULAR_HOOKS]`

本阶段不做。

处理方式：

- 不执行数据库查询。
- 不展示 `[POPULAR_HOOKS]` 标签。
- Prompt 中禁止依赖热门 hook。
- 后续支持 `social_post` 或热门内容库时再补。

### 步骤 4：设计内容结构 `[STRUCTURE]`

任务：设计 5-7 个章节，分配时长、密度和标题位置。

必须使用并保持顺序的 Section：

```text
[SECTION:hero]
[SECTION:features]
[SECTION:demo]
[SECTION:comparison]
[SECTION:summary]
[SECTION:references]
[SECTION:outro]
```

说明：

- `comparison` 可无声，但标记必须保留。
- `references` 可无声或 1 句话，但标记必须保留。
- 禁止省略、改名、乱序。

结构输出示例：

```text
[STRUCTURE]
[SECTION:hero] 20s, Density: Impact
[SECTION:features] 40s, Density: Standard
[SECTION:demo] 50s, Density: Standard
[SECTION:comparison] 35s, Density: Compact
[SECTION:summary] 25s, Density: Impact
[SECTION:references] 10s, Density: Standard
[SECTION:outro] 15s, Density: Impact
```

密度映射：

```text
hero=Impact
features=Standard
demo=Standard
comparison=Compact
summary=Impact
references=Standard
outro=Impact
```

时长建议：

- `short`: 60-120 秒
- `medium`: 180-300 秒
- `long`: 420-720 秒

### 步骤 5：生成 Hook `[HOOK]`

任务：生成 1-2 句话作为开头。

规则：

- 避免“发现秘密”“最好的”“顶尖”等陈词滥调。
- `personal` 使用第一人称。
- `company` 使用第三人称。
- `professional-casual` 专业但不僵硬，可适度使用“你”。
- 不直接复用用户原始请求原句。
- 如果用户提供 `heroOpening`，优先使用 `heroOpening` 作为 hero 开头。

输出标签：

```text
[HOOK] {hook_text}
```

### 步骤 6：生成正文 / 脚本 `[CONTENT]`

任务：根据结构和 hook 生成完整视频旁白。

规则：

- 必须严格使用 `[SECTION:xxx]` 标记。
- 每个标记后跟自然口语旁白。
- 换行使用 `\n`。
- 空段落表示无声，仅画面。
- `hero` 段必须以 `heroOpening` 或 hook 开头。
- `outro` 段必须以 `outroClosing` 或平台默认 CTA 结尾。
- 应用 TTS 数字格式规则。
- 遵循 `tone` 和 `verbosity`。

输出标签：

```text
[CONTENT] 完整脚本文本
```

### 步骤 7：后处理与格式化 `[OUTPUT]`

任务：校验格式、估算时长、组装最终 JSON。

时长估算：

- `zh-CN`: 字符数 ÷ 4.5
- `en-US`: 词数 ÷ 13

Warnings 规则：

- 如果估算时长小于目标时长下限，写入 warning。
- 如果估算时长大于目标时长上限，写入 warning。
- 如果搜索降级，写入 warning。
- 如果 Section 自动补全，写入 warning。

最终执行 Zod 校验，失败自动重试 1 次，仍失败则返回：

```json
{
  "status": "failed",
  "metadata": {...},
  "script": "",
  "warnings": ["生成结果格式校验失败"],
  "progress_log": [...]
}
```

## TTS 数字格式规则

必须保留阿拉伯数字：

- 年份/日期：`2025年`、`1月15日`、`2025年1月15日`
- 带单位数字：`18个月`、`3年`、`45天`、`2小时`、`2900万`、`5亿`、`300块`
- 百分比/小数：`15%`、`90%`、`-10%`、`1.2`、`3.5`
- 技术单位：`128GB`、`16核`、`4K`、`2TB`
- 小整数：`29`、`50`、`99`

必须转中文拼写：

- ISO 日期：`2025-01-15` 转 `2025年1月15日`
- 多段版本号：`v1.2.3` 转 `v一点二点三`
- 电话/身份证：`400-123-4567` 转 `四零零 一二三 四五六七`
- 无单位长整数：`3999999` 转中文或近似 `400万`

实现要求：

- 在 prompt 中明确约束。
- 在 TypeScript 后处理阶段增加正则修正。
- 对无法安全修正的内容写入 warning。

### 创建任务

`POST /api/content-generation/jobs`

职责：

- 校验登录态。
- 校验请求参数。
- 固定 `type = VIDEO_SCRIPT`。
- 写入 `ContentGenerationJob`。
- 返回 `jobId` 和 `status`。

### 查询详情

`GET /api/content-generation/jobs/:jobId`

职责：

- 只允许查询当前用户任务。
- 返回任务基础信息、阶段事件、最终结果。
- 阶段事件包含标签化 payload。

### SSE 进度接口

`GET /api/content-generation/jobs/:jobId/events`

第一版可选。如果实现成本较高，可以先用轮询，保留接口设计。

SSE 事件格式：

```text
event: progress
data: {"step":"prefs_loaded","ts":1715000000000,"message":"已加载用户偏好"}

event: progress
data: {"step":"research_done","ts":1715000002000,"message":"已收集 3 条关键信息"}

event: complete
data: {"final":{...完整JSON...}}
```

前端优先支持轮询；SSE 作为体验增强，不阻塞 MVP。

## TypeScript Worker 设计

不新增 Python worker。第一版在 `api-server` 内新增 TypeScript worker 脚本和服务层，复用现有 Prisma、配置、日志、Zod、Qwen/OpenAI-compatible HTTP 调用能力。

新增文件建议：

```text
api-server/src/workers/content-generation-worker.ts
api-server/src/modules/content-generation/content-generation-service.ts
api-server/src/modules/content-generation/content-generation-repository.ts
api-server/src/modules/content-generation/content-generation-generator.ts
api-server/src/modules/content-generation/content-generation-search.ts
api-server/src/modules/content-generation/content-generation-schemas.ts
api-server/src/modules/content-generation/content-generation-prompts.ts
api-server/src/modules/content-generation/content-generation-format.ts
```

`api-server/package.json` 新增脚本：

```json
{
  "content-generation:worker": "bun run src/workers/content-generation-worker.ts"
}
```

Worker 运行方式：

- 使用 Bun 常驻进程轮询 `ContentGenerationJob`。
- 每次领取一条 `PENDING` 或可重试任务。
- 使用 `workerId + lockedAt` 防止多进程重复处理。
- 阶段事件写入逻辑放在 service/repository 中，避免 route 和 worker 重复实现。
- LLM 超时或工具失败时最多重试 2 次。
- Zod 校验失败时自动重试生成 1 次。
- 第一版不做复杂指数退避；保留 `retryCount` 和 `nextRetryAt` 字段为后续扩展。

## 搜索服务

第一版需要满足“近期信息研究”，但不引入复杂数据库。

新增配置：

```yaml
search:
  provider: "duckduckgo"
  timeoutSeconds: 20
```

或使用环境变量：

```text
SEARCH_PROVIDER=duckduckgo
SEARCH_TIMEOUT_SECONDS=20
```

结果处理：

- 最多取 5 条候选。
- 至少抽取 3 条高质量内容；不足 3 条时继续执行，但在 `warnings` 标记。
- 摘要压缩为 3-5 个要点。
- 每条要点保留来源标题和 URL 到 `detailsJson.sources`。

降级策略：

- 搜索失败时不阻塞任务。
- 写入 `[RESEARCH]`，说明“网络搜索失败，使用模型基于通用知识生成低置信研究摘要”。
- 最终 `warnings` 追加搜索降级提示。

## LLM 生成服务

新增 TypeScript `ContentGenerationGenerator`：

- 使用 Qwen/OpenAI-compatible Chat Completion URL。
- 请求 `response_format: { type: "json_object" }`。
- 每一步使用 Zod Schema 严格校验。
- LLM 输出失败时记录 raw preview，最多重试 1 次。

建议配置：

```text
CONTENT_GENERATION_LLM_URL
CONTENT_GENERATION_LLM_MODEL
QWEN_API_KEY
QWEN_MOCK_MODE
```

建议在 `api-server/src/lib/qwen.ts` 增加通用 JSON 生成方法，避免在 worker 中重复写 HTTP 细节：

```ts
export async function generateJsonText(cfg: AppConfig["qwen"], params: {
  model: string;
  messages: Array<{ role: "system" | "user"; content: string }>;
  temperature: number;
}): Promise<unknown>;
```

mock 模式返回稳定假数据，保障本地开发和测试不依赖真实模型。

## 前端状态流

沿用 `bilibili-analysis-workspace.tsx` 的轮询思路：

1. 创建任务时清空旧结果。
2. 每 3 秒轮询任务详情。
3. `READY` 时停止轮询。
4. `FAILED` 时展示错误。
5. 页面首次加载时恢复最近任务或本地缓存任务。

## 格式化与校验

TypeScript Worker 在写回前执行本地校验：

- `status` 必须是 `completed` 或 `failed`。
- `metadata.topic` 非空。
- `metadata.platform` 与请求一致。
- `metadata.language` 与请求一致。
- `metadata.sections` 必须包含固定 Section，顺序正确。
- `script` 必须包含全部 `[SECTION:xxx]` 标记。
- `hook` 非空。
- `warnings` 是数组。
- `progress_log` 是数组。
- 不允许 Markdown 代码块包裹最终 JSON。
- 不输出 `social_content`、`suggested_date`、`image_prompts`。

Section 修复策略：

- 缺少可选内容但标记存在时允许空段。
- 标记缺失时触发重试。
- 重试后仍缺失，则自动补全空段并写入 warning。

## 风险与处理

### 搜索结果不足

处理：允许低置信降级，不阻塞生成，在 `warnings` 中标记。

### LLM 输出不稳定

处理：每一步独立 Zod 校验，失败重试 1 次，最终失败写入任务错误。

### Section 标记缺失

处理：优先重试；仍失败则自动补全空段并写入 warning。

### 任务耗时过长

处理：异步 Job + 轮询，不阻塞前端请求；SSE 作为后续体验增强。

### 需求范围膨胀

处理：第一版固定只做 `video_script`，不开放 `social_post`、图片、点数、发布时间。

### 前端解析标签困难

处理：主协议用结构化 `stageEvents.details`，标签只作为展示字段，不依赖纯文本解析。
