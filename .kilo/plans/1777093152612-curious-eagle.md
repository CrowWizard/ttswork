# 手机号注册登录改造计划

## 目标
将当前基于匿名 Cookie 的用户识别方式，升级为正式账号体系：
- 注册：手机号短信验证码注册，可选设置密码。
- 登录：支持手机号 + 密码、手机号 + 短信验证码。
- 替换原有以匿名 `AnonymousUser` 为主体的数据归属与鉴权逻辑。

## 当前实现证据
- 前端是 Next.js 15 单页应用，核心交互在 `components/voice-studio.tsx`，当前直接请求 `/api/voice/*` 与 `/api/tts`。
- 后端是 `api-server` 下的 Hono/Bun 服务，所有业务路由都调用 `ensureAnonymousUserCookie` 与 `ensureAnonymousUserRecord`。
- 数据模型当前只有 `AnonymousUser`，`VoiceEnrollment`、`TtsJob` 都直接外键到匿名用户。
- 当前仓库内没有现成的账号、密码、短信验证码、JWT/Session 体系，也没有短信服务配置项。
- 阿里云 `SendSmsVerifyCode` 文档显示：
  - 可由阿里云生成验证码并通过 `CheckSmsVerifyCode` 校验。
  - 也可业务端自生成验证码，但那样阿里云无法代校验。
  - 接口自带有效期、频控、重复发送策略等能力。

## 方案对比

### 方案 A：数据库会话 + HttpOnly Cookie（推荐）
- 服务端新增正式 `User`、`SmsVerification`/短信凭证、`Session` 等模型。
- 登录/注册成功后，服务端签发随机 session token，存数据库，浏览器仅保存 HttpOnly Cookie。
- 所有现有业务接口改为通过会话解析当前登录用户。

优点：
- 与当前 Cookie 使用习惯一致，前端改动相对小。
- 服务端可随时失效会话，便于登出、风控、封禁。
- 不依赖 JWT 刷新与吊销设计，适合当前单体后端。

缺点：
- 需要增加 `Session` 表与服务端会话查询。
- 若后续多端接入，需要再扩展设备管理策略。

### 方案 B：JWT/Refresh Token
- 注册/登录成功后返回 access token + refresh token。
- 业务接口改为 Bearer token 鉴权。

优点：
- 更适合未来开放 App/多端 API。
- 无状态 access token 读取快。

缺点：
- 当前项目没有现成 token 基础设施，改造面更大。
- refresh token、吊销、轮换、跨端退出会更复杂。
- 对当前浏览器单页场景收益不明显。

## 推荐决策
采用方案 A：数据库会话 + HttpOnly Cookie。

原因：
- 能最小化替换当前“用 Cookie 标识用户”的实现路径。
- 更容易把现在的匿名用户依赖切换成正式用户依赖。
- 对浏览器端最稳妥，也便于后续补充短信登录、密码登录。

## 已确认产品决策
- 允许直接废弃旧匿名数据，只保留新账号体系，不做匿名数据迁移与绑定。
- 不需要用户名能力，本次账号唯一标识仅为手机号。
- 短信注册成功但未设置密码的用户，后续允许仅通过短信登录。

## 需要改造的主要模块

### 1. 数据模型
计划将 `AnonymousUser` 迁移为正式账号体系，新增或调整：
- `User`
  - `id`
  - `phoneNumber`（唯一）
  - `passwordHash`（可空，支持“仅短信注册”）
  - `phoneVerifiedAt`
  - `createdAt` / `updatedAt`
  - `activeVoiceEnrollmentId`
- `Session`
  - `id`
  - `userId`
  - `tokenHash`
  - `expiresAt`
  - `createdAt`
  - `lastSeenAt`
- `SmsLoginChallenge` 或 `SmsVerification`
  - 记录手机号、场景、阿里云回执标识、过期时间、校验状态。
- 业务表外键调整：
  - `VoiceEnrollment.userId` 改指向 `User`
  - `TtsJob.userId` 改指向 `User`

### 2. 鉴权与会话基础设施
- 新增认证常量、Cookie 名、会话过期时间配置。
- 新增密码哈希/校验工具。
- 新增当前用户解析中间件或公共方法，替代 `ensureAnonymousUserCookie`。
- 新增登录态检查与登出能力。

### 3. 短信验证码能力
- 新增阿里云短信认证客户端封装。
- 推荐采用阿里云生成验证码 + 服务端调用校验接口，不在本地存明文验证码。
- 增加发送频控、场景区分、错误码映射。
- 增加配置项：AccessKey、签名、模板、验证码长度、有效期等。

### 4. 认证接口
计划新增：
- `POST /api/auth/sms/send`
  - 场景：`register` / `login`
- `POST /api/auth/register`
  - 参数：手机号、短信验证码、可选密码
- `POST /api/auth/login/password`
  - 参数：手机号、密码
- `POST /api/auth/login/sms`
  - 参数：手机号、短信验证码
- `POST /api/auth/logout`
- `GET /api/auth/me`

### 5. 现有业务接口改造
以下现有路由都需要从“匿名用户”改为“当前登录用户”：
- `GET /api/voice/profile`
- `POST /api/voice/enroll`
- `GET /api/voice/enrollments/:id/audio`
- `POST /api/voice/enrollments/:id/invalidate`
- `GET /api/tts`
- `POST /api/tts`
- `GET /api/tts/:jobId/download`

### 6. 前端页面与交互
- 在 `components/voice-studio.tsx` 外层增加登录门禁，未登录先显示认证面板。
- 增加注册/登录 Tab：
  - 手机号短信注册
  - 手机号 + 密码登录
  - 手机号短信登录
- 增加短信发送倒计时、验证码输入、错误提示、登录态刷新。
- 登录成功后拉取 `me` 与业务数据。
- 登出后清空前端状态并回到认证页。

## 详细实施计划

### 阶段 1：数据模型与鉴权骨架
1. 在 `prisma/schema.prisma` 中删除匿名账号模型依赖，改为正式 `User` + `Session` + `SmsVerification` 体系。
2. 将 `VoiceEnrollment.userId`、`TtsJob.userId` 外键切换到 `User`。
3. 保持 `activeVoiceEnrollmentId` 语义不变，挂在 `User` 上，减少既有业务逻辑重写量。
4. 预留短信场景字段，至少区分 `register` 与 `login`，避免验证码串用。

### 阶段 2：服务端认证能力
1. 在 `api-server/src/lib` 新增认证模块：
   - 会话 token 生成、摘要存储、过期校验、登出失效。
   - 密码哈希与比对。
   - 当前登录用户解析，替代 `ensureAnonymousUserCookie` / `ensureAnonymousUserRecord`。
2. 在配置层新增认证与阿里云短信所需环境变量：
   - 会话 Cookie 名、过期时间。
   - 阿里云 AccessKey、Secret、SignName、TemplateCode、有效期、频控间隔。
3. 统一未登录错误返回，现有业务接口全部改为显式 `401`/`403`。

### 阶段 3：短信发送与校验
1. 封装阿里云短信认证客户端，优先采用“阿里云生成验证码 + 服务端校验”的链路。
2. 为发送接口加入手机号格式校验、场景校验与频控错误映射。
3. 注册时校验验证码后创建用户；登录时校验验证码后直接创建会话。
4. 不在本地持久化明文验证码，只保存必要的业务追踪字段与状态。

### 阶段 4：认证 API
1. `POST /api/auth/sms/send`
   - `register`：已注册手机号禁止继续发注册验证码。
   - `login`：未注册手机号禁止继续发登录验证码。
2. `POST /api/auth/register`
   - 手机号唯一校验。
   - 验证码通过后创建用户。
   - 有密码则写入 `passwordHash`，无密码则允许后续仅短信登录。
   - 注册成功直接创建会话并登录。
3. `POST /api/auth/login/password`
   - 仅支持手机号。
   - 若用户未设置密码，返回明确错误，引导使用短信登录。
4. `POST /api/auth/login/sms`
   - 校验通过后创建新会话。
5. `GET /api/auth/me` 与 `POST /api/auth/logout`
   - 分别用于前端启动时恢复登录态与主动退出。

### 阶段 5：业务接口迁移
1. 将下列路由统一改为从会话中解析 `currentUser`：
   - `GET /api/voice/profile`
   - `POST /api/voice/enroll`
   - `GET /api/voice/enrollments/:id/audio`
   - `POST /api/voice/enrollments/:id/invalidate`
   - `GET /api/tts`
   - `POST /api/tts`
   - `GET /api/tts/:jobId/download`
2. 所有用户过滤条件从匿名 `id` 切换为正式 `user.id`。
3. 删除匿名用户兜底创建逻辑，未登录直接拒绝访问。

### 阶段 6：前端认证门禁与交互
1. 在 `components/voice-studio.tsx` 前增加 `auth/me` 启动检查。
2. 未登录时展示认证卡片，包含：
   - 手机号短信注册（验证码、可选密码）
   - 手机号密码登录
   - 手机号短信登录
3. 登录成功后再加载 profile 与 TTS 历史。
4. 对“未设置密码”的账号，在密码登录失败提示中明确说明需改用短信登录。
5. 增加登出按钮与登录态切换后的本地状态清理。

### 阶段 7：清理与回归
1. 删除不再使用的匿名常量、匿名 session 工具与相关类型。
2. 同步更新 README / 环境变量示例中的认证与短信配置说明。
3. 执行 lint、typecheck、前端 build、`api-server` build，并修复回归问题。

## 数据迁移策略
- 已确认不保留旧匿名数据。
- 本次改造按“新账号体系完全替换旧匿名体系”实施。
- Prisma 迁移可直接移除匿名用户关系，不需要设计历史数据绑定脚本。
- 风险集中在开发与测试环境中的旧数据失效，需要在交付说明中明确告知。

## 验收标准
1. 未登录访问现有语音与 TTS 业务接口时，服务端返回未授权错误，不再自动创建匿名用户。
2. 用户可通过手机号短信验证码完成注册，且可选择是否同时设置密码。
3. 已设置密码的用户可通过手机号 + 密码登录。
4. 未设置密码的用户无法使用密码登录，但可通过手机号短信登录。
5. 登录成功后可正常完成查看 profile、建声、作废 active voice、文本转语音、下载音频全流程。
6. 登出后再次访问受保护接口会失去访问权限。
7. 工程校验通过：`npm run lint`、`npm run typecheck`、`npm run build`，以及 `api-server` 构建成功。
