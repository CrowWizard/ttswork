# 修复实施计划

## 目标
基于 `.kilo/reports/1777108306467-clever-river-audit.md` 中已确认的问题，完成一轮以可访问性、主题系统、响应式和最小性能改造为主的前端修复。

本轮目标不是做完整品牌重设计，而是在不引入大规模重写和不依赖缺失的 `PRODUCT.md` / `DESIGN.md` 前提下，把 P1 问题全部落地修复，并消化对代码与体验收益最高的 P2 问题。

## 实施范围
- `components/voice-studio.tsx`
- `app/register/page.tsx`
- `app/settings/page.tsx`
- `app/globals.css`
- `tailwind.config.ts`
- 视情况新增极少量复用组件文件，例如：
  - `components/ui/status-message.tsx`
  - `components/ui/form-field.tsx` 或更轻量的状态/容器基元

## 不做事项
- 不补 `PRODUCT.md`、`DESIGN.md`，因为当前用户要求是先修代码问题。
- 不做真正 dark mode 切换器；但会解除硬编码 `color-scheme: light`，让系统主题与后续扩展不被锁死。
- 不做大规模信息架构重构，不拆成很多小组件；只做最小必要拆分，优先降低热点耦合与重复渲染面。
- 不改后端 API 协议。

## 方案对比
### 方案 A：最小可用修复（推荐）
内容：
- 修完全部 P1
- 修 3 个高价值 P2：`recordError` 语义拆分、录音计时降频/隔离、移动端触控与截断问题
- 对 `VoiceStudio` 做最小拆分，而不是完全模块化重构

优点：
- 风险小，能较快改善审计分数
- 不容易引入首页登录/建声/TTS 流程回归
- 与当前代码体量、缺失设计文档的现状匹配

缺点：
- 视觉层面的“模板感”只能缓解，不能彻底重塑
- `VoiceStudio` 仍会偏大，只是比现在更可维护

### 方案 B：中等幅度重构
内容：
- 在方案 A 基础上，进一步拆解首页为多个独立子组件并重做页面视觉层级

优点：
- 长期维护性更好
- Anti-Patterns 分数改善空间更大

缺点：
- 回归面明显扩大
- 在缺失产品/设计基线时，容易做成新的主观版本

### 结论
采用 **方案 A**。先把“可访问性阻断 + 主题基础设施 + 高收益结构修复”落地，再根据结果决定是否继续做更大重构。

## 实施步骤

### 1. 建立语义化主题基础
目标：先收敛颜色、表面层级、焦点样式的基础设施，避免后续页面逐点硬改。

修改方向：
- 在 `tailwind.config.ts` 中把现有原始颜色升级为语义 token，至少补齐：
  - `surface`
  - `surface-muted`
  - `surface-elevated`
  - `border-subtle`
  - `text-primary`
  - `text-secondary`
  - `text-muted`
  - `action-primary`
  - `action-primary-hover`
  - `danger`
  - `danger-surface`
  - `success`
  - `success-surface`
  - `warning`
  - `warning-surface`
  - 更克制的卡片阴影 token
- 在 `app/globals.css`：
  - 移除 `:root { color-scheme: light; }` 的硬锁定
  - 定义统一的 `focus-visible` 样式
  - 统一 body 字体栈，替换 `Arial, Helvetica, sans-serif`
  - 保留现有浅色基调，但降低“奶油渐变 + 白卡 + 柔阴影”模板感

验收标准：
- 全局不再依赖 `color-scheme: light`
- 页面主要视觉值不再直接写死为旧 token 组合
- 键盘焦点在输入框和按钮上清晰可见

### 2. 抽出统一状态消息与焦点/输入样式
目标：一次性解决多页重复的状态反馈语义缺失与表单焦点弱问题。

修改方向：
- 新增轻量状态消息组件，统一支持：
  - `type: error | success | info | warning`
  - `role="alert"` 或 `role="status"`
  - `aria-live="assertive" | "polite"`
- 在 `voice-studio`、`register`、`settings` 三页替换现有散落的消息块
- 输入类控件统一去掉 `outline-none + focus:border-*` 这种弱焦点组合，改成全局共享的 focus ring / ring offset / border 组合

验收标准：
- 登录、注册、密码设置/修改、建声、TTS 的成功/失败消息全部具备可读语义
- 三个页面不再重复维护不同风格的消息块
- 主要输入控件都有稳定的 `focus-visible`

### 3. 修复录音主交互的键盘可达性
目标：消除首页核心任务的 P1 阻断。

修改方向：
- 将当前“按住录音”交互升级为双模式：
  - Pointer 模式仍支持按住开始、松开结束
  - Keyboard 模式改为按一次开始、再按一次结束，或对 `Space` / `Enter` 做显式切换逻辑
- 为录音按钮补充：
  - `aria-pressed` 或更准确的状态表达
  - 更明确的辅助说明文案
  - 录音中、上传中、禁用中的语义状态文本
- 审查 `startRecording` / `stopRecording` 的调用时机，避免键盘触发与鼠标触发产生重复 stop/start

关键注意：
- 不能破坏现有 pointer 用户“按住说话”的直觉流程
- 必须防止录音中再次触发 start 造成状态错乱

验收标准：
- 键盘用户可完整完成开始录音与结束录音
- pointer 路径行为与当前一致或更稳定
- 录音按钮状态文案与辅助技术语义一致

### 4. 拆分 `recordError`，统一录音/TTS 状态模型
目标：修掉语义混乱与状态复用问题，为可访问性和后续维护打底。

修改方向：
- 将 `recordError` 拆分为至少两类：
  - 错误态，例如 `workspaceError` / `recordError`
  - 成功态/提示态，例如 `workspaceNotice` / `recordStatus`
- 建声成功、作废成功、TTS 失败、加载失败分别落到正确状态槽位
- 配合第 2 步的统一消息组件渲染

验收标准：
- 不再出现“成功信息走 error 状态变量”的情况
- 不同语义使用不同颜色、不同 ARIA 角色

### 5. 对 `VoiceStudio` 做最小拆分并降低录音计时开销
目标：降低热点重渲染范围，但避免大手术。

修改方向：
- 从 `components/voice-studio.tsx` 中优先切出高独立度部分：
  - 账号头部区域
  - 认证面板
  - 录音建声区
  - TTS 区
- 录音计时改造二选一，优先选择风险更低者：
  1. 将计时显示拆到小组件内部，父组件只持有录音状态
  2. 保持状态在父组件，但把刷新频率从 `100ms` 降到 `250ms` 或 `500ms`
- 推荐先做：**降频 + 小范围拆分**，不做彻底状态管理重构

验收标准：
- `VoiceStudio` 明显缩短，职责边界更清楚
- 录音计时不再以 100ms 频率驱动整页大范围更新
- 登录、建声、TTS 既有行为保持一致

### 6. 修复移动端触控与信息承载问题
目标：消化高价值响应式 P2 问题。

修改方向：
- 放大 active voice 作废按钮到至少接近 `44x44`
- 对 active voice 值提供：
  - 更大的可用宽度，或
  - 窄屏换行展示，避免单纯 `truncate` 吃掉关键信息
- 设置页账号信息在窄屏下从“左右对齐”改为“上下堆叠优先”，避免大字号压缩内容
- 复查验证码区、操作按钮区在窄屏下的间距一致性

验收标准：
- 窄屏下无关键内容被过度截断
- 触控按钮尺寸和误触容忍度改善
- 设置页在手机宽度下阅读成本下降

### 7. 缓解模板化视觉与弱状态表达
目标：在不做彻底品牌重设计的前提下，把明显模板感往下压一档。

修改方向：
- 收敛顶部大卡与内容卡的视觉层级：
  - 减少重复的大圆角等级
  - 减少不必要的 `backdrop-blur` / 半透明白底
  - 让不同区域在背景、边框、阴影强度上有层次差异
- 强化加载态/空态：
  - `active voice` 空态增加下一步提示
  - 设置页加载态改为更结构化的占位
  - TTS 历史为空时补任务导向文案

验收标准：
- 三个页面不再像同一模板复制
- 空态/加载态更可理解，不只是一句“加载中...”

## 验证计划
必须本地自动验证，不依赖人工。

### 静态验证
- `npm run lint`
- `npm run typecheck`
- `npm run build`

### 代码级人工核对点
- 键盘可完整操作录音按钮
- 成功态与错误态消息具备正确 `role` / `aria-live`
- 输入与按钮焦点在浅色背景中清晰可见
- `recordError` 已被语义化拆分，不再承载成功提示
- `VoiceStudio` 体量和职责较改造前明显收敛
- 手机宽度下 active voice、设置页信息和验证码区域无明显挤压

## 风险与缓解
### 风险 1：录音交互改造引入重复 start/stop
缓解：
- 明确区分 pointer 与 keyboard 触发路径
- 用现有 `recording` / `enrolling` / `invalidating` 守卫避免重复进入

### 风险 2：主题 token 重命名造成页面样式回退
缓解：
- 先新增语义 token，再逐步替换旧 class
- 一轮内避免删除所有旧 token，确认页面落稳后再清理

### 风险 3：`VoiceStudio` 拆分过多导致状态穿透复杂
缓解：
- 只拆纯展示或边界清晰的区块
- 保持核心流程状态仍在父组件，不引入新状态管理库

### 风险 4：视觉去模板化过头，偏离现有产品气质
缓解：
- 保留当前浅色、温和、工具型基调
- 只调整层级与重复度，不做激进风格跳变

## 推荐执行顺序
1. 主题 token 与全局焦点样式
2. 统一状态消息组件
3. 录音按钮键盘可达性修复
4. `recordError` 语义拆分
5. `VoiceStudio` 最小拆分与计时优化
6. 移动端触控与信息承载修复
7. 视觉层级与空态/加载态收口
8. `lint`、`typecheck`、`build` 全量验证

- Theming 至少从 `1/4` 提升到 `2/4` 或 `3/4`
- Performance 至少从 `2/4` 提升到 `3/4`
- Responsive Design 稳定在 `3/4` 并接近 `4/4`
- Anti-Patterns 从 `1/4` 提升到 `2/4`

## 实施建议
