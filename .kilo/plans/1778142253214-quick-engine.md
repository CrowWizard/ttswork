# 视频分析功能扩展计划

## 目标

基于 `tmp/video_analyzer_v3.py` 的 V3 示例改造现有 B 站视频分析功能。保留现有核心展示：

1. 视频基础信息
2. 内容摘要
3. 分段结构

新增示例中的传播包装、脚本结构、语义层、内化总结、元数据指标等分析维度，并同步修改前端界面、数据库/API、`video-analysis-worker`。

## 硬性约束

- 不修改当前 worker 使用的 LLM、model、API key 来源和基础调用配置。
- 不切换到示例文件中的 `LLM_BASE_URL = "https://api.deepseek.com/v1"`、`LLM_MODEL = "deepseek-chat"` 或 `AsyncOpenAI` 客户端。
- 继续使用现有 `WorkerConfig`：`video_analysis_llm_url`、`video_analysis_llm_model`、`qwen_api_key`、`qwen_mock_mode`。
- 可借鉴示例的三阶段 prompt、Pydantic 结构、压缩策略和后处理计算，但实现仍落在现有 worker 架构内。
- 继续兼容 PostgreSQL 9.x，分析对象/数组仍以 `String?` 存储 JSON 文本，不引入 `json/jsonb`。
- 新增数据库字段全部可空，避免破坏历史任务。
- 完成实现后更新 `operations-log.md` 并执行本地验证闭环。

## 重新分析后的示例结论

`tmp/video_analyzer_v3.py` 现在是完整 V3 示例，包含以下关键结构：

- 数据模型层：
  - `Paragraph` / `ParagraphsWrapper`：长字幕压缩用的段落摘要。
  - `VideoHealthCard`：视频健康卡片。
  - `PackagingAnalysis`：标题、封面、心理触发和包装层分析。
  - `StructureExtract`：脚本结构层，包含钩子、叙事弧、结构块、金句、CTA、逻辑流。
  - `SemanticAnalysis`：语义传播层，包含心理触发、修辞、语气、网感词、互动、知识密度、认知负荷、情绪曲线。
  - `Internalization`：内化总结，包含核心信息、巧妙设计、优化建议。
  - `MetadataJSON`：指标与风险，包含钩子分、留存风险点、金句数、互动数、认知负荷分布等。
  - `FinalReport`：最终报告，字段为 `health_card`、`packaging`、`script_layer`、`semantic_layer`、`summary`、`metadata_json`。
- Prompt 示例层：
  - `STEP1_EXAMPLE` / `STEP1_CONSTRAINTS`：结构抽取。
  - `STEP2_EXAMPLE` / `STEP2_CONSTRAINTS`：包装层 + 语义层。
  - `STEP3_EXAMPLE` / `STEP3_CONSTRAINTS`：最终报告汇总。
- 流水线层：
  - `generate_paragraph_summary()`：长字幕先按约 60 秒生成段落摘要。
  - `build_compressed_text()`：保留开头 30 秒、结尾 30 秒和中间段落摘要。
  - `step1_extract_structure()`：提取脚本结构。
  - `step2_analyze_semantic_packaging()`：提取包装和语义。
  - `step3_generate_report()`：生成最终报告并做代码可确定的元数据后处理。
- 需要注意的示例问题：
  - 示例仍包含自己的 LLM 配置和 `AsyncOpenAI`，实现时不能照搬。
  - `StructuralBlocksOutput` 字段类型是 `Optional[str]` / `List[str]`，但示例 `step3_generate_report()` 里使用了 `sb.hook.start` 这类对象访问，实际实现应按字符串处理，不能复制该 bug。
  - 示例依赖带时间轴字幕，当前 worker 的字幕和 ASR 文本需要补足时间轴输入能力。

## 当前项目现状

- 前端：`components/bilibili-analysis/bilibili-analysis-workspace.tsx`

  - 当前 result 字段：`summary`、`structureSections`、`highlights`、`copySuggestions`、`modelName`、`promptVersion`。
- API：`api-server/src/routes/video-analysis.ts`
  - 当前 `toVideoAnalysisResultDto()` 只解析旧字段。
- 数据库：`api-server/prisma/schema.prisma`
  - `VideoAnalysisJob` 当前存储 `summary`、`structureSections`、`highlights`、`copySuggestions`。
- Worker：`video-analysis-worker/services/analyzer.py`
  - 当前是单次 LLM 分析。
  - 当前 schema 只允许旧四个字段，`extra="forbid"`。
  - 当前 mock、DB 写回、测试只覆盖旧字段。
- 时间轴：
  - `SubtitleService.fetch_text()` 当前只返回纯文本，丢弃 B 站字幕 `from` / `to`。
  - `AsrService._extract_text()` 当前只返回纯文本，未保留 DashScope `sentences` 的时间。

## 推荐方案

采用“显式顶层字段 + 三阶段 worker 分析”的方案。

### 为什么不只做单次 LLM

示例已经明确把结构层、包装/语义层、最终报告层拆开，并通过代码后处理计算 `hook_score`、`golden_quote_count`、`interaction_count`、`cognitive_load_distribution`。如果仍要求单次 LLM 一次性输出完整大 JSON，风险是：

- 输出长度更大，格式失败概率更高。
- 难以复用示例中的代码后处理逻辑。
- 结构、包装、语义三个任务互相干扰，字段质量更不稳定。

因此推荐在现有 `AnalyzerService.analyze()` 内部实现三阶段调用，但不改变 LLM endpoint/model/key。

### 数据库存储方案

采用显式字段列，而不是单个聚合 `analysisReport` 字段。

优点：

- API 契约清晰，前端类型明确。
- 每个分析维度可独立空态展示。
- 便于保留旧字段并兼容历史任务。
- 仍与当前 `structureSections`、`highlights`、`copySuggestions` 的 JSON 字符串存储方式一致。

代价：

- 需要同步修改 schema、API、worker、前端类型。
- 后续字段大改时还需迁移。

## 字段映射

### 旧字段继续保留

- `summary`: string
  - 展示为“内容摘要”。
  - 由 `health_card.one_line_summary`、`summary.core_message`、`summary.clever_design` 组合生成用户可读摘要。
  - 不能改成对象，避免破坏旧前端兼容。
- `structureSections`: JSON string
  - 展示为“分段结构”。
  - 保持数组 `{ title, startSeconds, endSeconds, summary }`。
  - 从段落摘要 `ParagraphsWrapper.paragraphs` 优先转换；若未触发长字幕压缩，则从时间轴文本按 60 秒或从 `script_layer.narrative_arc` 辅助生成。
- `highlights`: JSON string
  - 保留旧“金句 / 爆点”。
  - 从 `script_layer.quotes` 转换为 `{ quote, reason, timestampSeconds }`。
- `copySuggestions`: JSON string
  - 保留旧“可复用文案建议”。
  - 从 `packaging.title_formulas`、`packaging.title_hook_words`、`summary.optimization` 转换生成。

### 新增数据库字段

- `healthCard`: JSON string
  - 对应 `FinalReport.health_card`。
  - 内部字段保持示例 snake_case：`one_line_summary`、`core_keywords`、`has_hook`、`has_cta`、`hook_and_cta_quotes`。
- `packagingAnalysis`: JSON string
  - 对应 `FinalReport.packaging`。
  - 内部字段保持示例 snake_case。
- `scriptAnalysis`: JSON string
  - 对应 `FinalReport.script_layer`。
  - 内部字段保持示例 snake_case。
- `semanticAnalysis`: JSON string
  - 对应 `FinalReport.semantic_layer`。
  - 内部字段保持示例 snake_case。
- `internalizationSummary`: JSON string
  - 对应 `FinalReport.summary`。
  - 内部字段保持示例 snake_case。
- `metadataJson`: JSON string
  - 对应 `FinalReport.metadata_json`。
  - 内部字段保持示例 snake_case。

说明：数据库和 API 顶层字段使用项目既有 camelCase，JSON 对象内部字段保持示例 snake_case，降低模型和 prompt 改写成本。

### 4. Worker 三阶段分析

- 修改 `video-analysis-worker/services/analyzer.py`：

    1. 输入 `title`、`transcript_text`、`timeline_text`、`duration_seconds`、`cover_url` 或封面描述。

## 风险与处理

- 三阶段 LLM 调用耗时更长：保留现有超时配置，必要时只在格式失败或超时时把任务标记为稳定错误，不静默降级成空 schema。
- 示例里 `schema()` 空构造兜底不适合当前 worker：当前实现应继续失败即抛 `AnalysisResultFormatError`，避免写入不完整报告。
- 新字段依赖时间轴：优先补齐字幕/ASR 时间轴文本；缓存命中旧纯文本时允许保守估计时间。
- 示例 `StructuralBlocksOutput` 后处理存在类型访问 bug：实现时按字符串范围处理。
- 历史任务兼容：旧任务新增字段为空，但旧摘要、分段、金句、文案建议仍可展示。
- 前端信息过载：按“判断视频价值 -> 理解结构 -> 复用传播机制”的顺序展示，不把全部字段平铺。
