# B站视频分析异步任务链路分阶段实施计划

## 背景与目标
- 当前 `/video-analysis` 页面已经具备最小输入区与结果区骨架，但尚未接入真实后端分析流程。
- 已确认该模块不支持 `anonymousUserId`，必须要求登录后使用。
- 已确认采用弱缓存：每次分析都新建 `VideoAnalysisJob`，但允许复用同一个 `VideoSource`。
- 已确认当视频没有字幕时，需要自动走音频转写 ASR。
- 已确认真正的分析执行更适合由 Python 程序承担，而不是继续堆叠在现有 TypeScript API 内。

## 关键决策

### 1. 任务中转方案
- 第一阶段采用 `Postgres + job table + Python poller`。
- 理由：当前仓库已经稳定依赖 Postgres，且前端查询本就以数据库状态为真相源；先不额外引入 MQ，可显著降低首期复杂度与排障成本。

### 2. 长期演进方向
- 若后续出现多阶段 worker、并发任务量上升、跨语言消费者增加等需求，再升级为 `NATS JetStream`。
- 当前不采用 Celery，原因是现有系统边界为 `TS API + Python Worker`，Celery 对跨语言生产者不够自然。
- 当前也不建议首发引入 `Dapr`。理由是当前核心问题仅是“任务投递与执行解耦”，尚未进入需要统一 `pub/sub + state store + bindings + service invocation + workflow` 的平台化阶段；若未来系统明显服务化，再评估 `Dapr + NATS` 组合。

### 3. 数据建模方向
- 采用两层模型：
  - `VideoSource`：全局复用的视频源与内容提取缓存。
  - `VideoAnalysisJob`：按用户创建的单次分析任务快照。
- `VideoSource` 建议按 `normalizedBvid` 全局唯一复用，不挂 `userId`。
- `VideoAnalysisJob` 必须挂 `userId`，每次请求都新建。

## 推荐数据结构

### VideoSource
- 职责：保存视频基础信息、字幕获取状态、最终分析转写文本。
- 建议字段：
  - `id`
  - `platform`：固定 `BILIBILI`
  - `inputType`：`URL | BV`
  - `inputValue`
  - `normalizedBvid`：唯一键
  - `normalizedUrl`
  - `title`
  - `authorName`
  - `authorMid`
  - `coverUrl`
  - `durationSeconds`
  - `publishTime`
  - `subtitleStatus`：`PENDING | READY | UNAVAILABLE | FAILED`
  - `transcriptStatus`：`PENDING | READY | FAILED`
  - `transcriptSource`：`SUBTITLE | ASR | null`
  - `subtitleText`
  - `transcriptText`
  - `fetchErrorMessage`
  - `createdAt`
  - `updatedAt`

### VideoAnalysisJob
- 职责：保存某个用户的一次分析任务、处理状态与结构化结果。
- 建议字段：
  - `id`
  - `userId`
  - `videoSourceId`
  - `status`：`PENDING | PROCESSING | READY | FAILED`
  - `errorMessage`
  - `summary`
  - `structureSections`：JSON
  - `highlights`：JSON
  - `copySuggestions`：JSON
  - `modelName`
  - `promptVersion`
  - `workerId`
  - `lockedAt`
  - `retryCount`
  - `nextRetryAt`
  - `createdAt`
  - `updatedAt`
  - `completedAt`

## API 范围
- `POST /api/video-analysis/jobs`
- `GET /api/video-analysis/jobs/:jobId`
- `GET /api/video-analysis/jobs`
- `GET /api/video-analysis/workspace`

## 分阶段计划

### 阶段 1：数据模型与只读接口骨架
目标：先把数据库真相源和前端查询接口稳定下来，不急着接 Python 执行器。

实施内容：
- 在 Prisma schema 中新增：
  - `VideoSource`
  - `VideoAnalysisJob`
  - 相关枚举，如 `VideoAnalysisJobStatus`、`VideoSubtitleStatus`、`VideoTranscriptStatus`
- 为 `VideoSource.normalizedBvid` 建唯一索引。
- 为 `VideoAnalysisJob` 建索引：
  - `userId + createdAt`
  - `videoSourceId + createdAt`
  - `status + createdAt`
- 设计并实现 4 个接口的响应 DTO，而不是直接暴露 ORM 原始结构。
- 此阶段先让 `POST /api/video-analysis/jobs` 只完成：
  - 登录校验
  - 输入校验
  - 解析 BV / URL
  - 查找或创建 `VideoSource`
  - 创建 `VideoAnalysisJob`
- `GET` 接口先基于数据库读取返回空态 / 基础状态，不要求任务真正被执行。

验收标准：
- 前端可以成功提交任务并拿到 `jobId`。
- 前端可以查询到自己刚提交的任务。
- 列表和工作台接口只返回当前登录用户的数据。
- 数据表与索引设计稳定，可支撑后续 Python worker 接入。

### 阶段 2：Python Worker 最小闭环
目标：让任务真正跑起来，完成“抓视频 -> 字幕优先 -> 无字幕转写 -> 结构化分析 -> 回写结果”的闭环。

实施内容：
- 新建独立 Python worker 程序。
- worker 周期性从数据库领取 `PENDING` 任务。
- 抢任务时必须原子化更新为 `PROCESSING`，同时写入：
  - `workerId`
  - `lockedAt`
- Worker 执行顺序：
  - 读取 `VideoAnalysisJob`
  - 关联加载 `VideoSource`
  - 获取视频基础信息
  - 优先尝试抓字幕
  - 若字幕不可用，则拉音频做 ASR
  - 将最终文本写回 `VideoSource.transcriptText`
  - 调用 LLM 生成结构化分析结果
  - 回写 `summary / structureSections / highlights / copySuggestions`
  - 更新 `VideoAnalysisJob.status = READY`
- 如果失败：
  - 更新 `VideoAnalysisJob.status = FAILED`
  - 记录 `errorMessage`

验收标准：
- 至少 1 个公开视频可以完整跑通分析链路。
- `GET /api/video-analysis/jobs/:jobId` 能看到完整结构化结果。
- 有字幕视频走字幕链路，无字幕视频走 ASR 链路。
- 同一视频重复分析时复用原有 `VideoSource`，但仍新增 `VideoAnalysisJob`。

### 阶段 3：任务可靠性与可恢复能力加固
目标：避免任务丢失、重复执行、长时间卡死和难以排障。

实施内容：
- 增加任务锁超时回收逻辑：
  - `PROCESSING` 且 `lockedAt` 超时的任务允许重新领取。
- 增加退避重试机制：
  - 使用 `retryCount`
  - 使用 `nextRetryAt`
- 增加链路日志与诊断上下文：
  - 输入类型
  - `normalizedBvid`
  - 字幕是否命中
  - 是否走 ASR
  - 分析耗时
  - worker 标识
- 对错误分类：
  - 视频不存在 / 失效
  - 字幕抓取失败
  - 音频拉取失败
  - ASR 失败
  - LLM 失败
- 对 `GET /workspace` 增加聚合信息，例如：
  - 最近任务
  - 当前进行中任务
  - 最近失败任务摘要

验收标准：
- Worker 异常退出后，卡住的任务可回收重试。
- 同一任务不会被多个 worker 长时间重复消费。
- 常见失败场景能在数据库状态和日志中快速定位。

### 阶段 4：前端接入真实任务状态
目标：让 `/video-analysis` 页面从静态骨架变成真实任务界面，但不做大规模 UI 重构。

实施内容：
- 页面提交时调用 `POST /api/video-analysis/jobs`。
- 页面持有 `currentJobId`，轮询 `GET /api/video-analysis/jobs/:jobId`。
- 根据状态渲染：
  - `PENDING`：已创建任务，等待处理
  - `PROCESSING`：正在抓取 / 转写 / 分析
  - `READY`：展示完整结果
  - `FAILED`：展示错误态与重试引导
- 可视情况接入 `GET /api/video-analysis/workspace` 作为首屏初始化数据源。

验收标准：
- 页面可以提交真实分析任务。
- 页面可看到处理中状态，而不是长时间空白。
- 页面结果区与现有 5 个分区准确映射。

### 阶段 5：可选升级到 NATS JetStream
目标：当 DB 轮询模型开始成为瓶颈时，再引入真正的消息投递层。

触发条件：
- 任务量明显上升。
- 需要拆分多个 worker 阶段。
- 需要跨语言扩展更多消费者。
- 单纯依赖 DB 轮询已经造成明显延迟或锁竞争。

实施内容：
- `POST /api/video-analysis/jobs` 在入库成功后发布消息到 NATS JetStream。
- Python worker 从 JetStream 消费任务消息。
- 数据库继续保留为任务状态真相源。
- NATS 只负责投递与唤醒，不负责最终状态查询。

补充说明：
- 若系统后续不仅需要消息队列，还需要统一抽象状态存储、服务调用、定时工作流、外部绑定等能力，再评估是否在 `NATS JetStream` 之上增加 `Dapr` 运行时。
- 在仅有视频分析 worker 的阶段，不建议为了抽象一致性提前引入 `Dapr sidecar`，否则会额外增加本地调试、部署与排障复杂度。

验收标准：
- 投递层与状态层分离。
- 前端无需感知 MQ 的存在，仍只依赖数据库聚合接口。
- Worker 横向扩展时延迟和资源占用优于 DB 轮询方案。

## 关键实现注意点
- 不要在首期把 Celery 引入到 `TS API + Python Worker` 的跨语言边界中。
- 不要让消息系统替代数据库成为任务状态真相源。
- 不要把 `GET` 接口做成 ORM 原始记录直出，统一通过 DTO / ViewModel 组装返回。
- `VideoSource` 应只表达“视频内容提取与缓存”，不要混入某次分析结果。
- `VideoAnalysisJob` 应只表达“某次分析快照”，不要把可复用内容缓存塞回任务表。

## 每阶段建议验证

### 阶段 1
- API 参数校验
- 登录态校验
- Prisma generate / typecheck

### 阶段 2
- 真实公开视频分析 1 条
- 有字幕 / 无字幕 两类样例验证
- 任务从 `PENDING` 到 `READY` 的完整状态迁移

### 阶段 3
- 人工制造 worker 中断
- 验证锁回收与重试
- 验证失败日志可追踪

### 阶段 4
- 前端提交 + 轮询 + 结果渲染
- 移动端空态、处理中、失败态不溢出

### 阶段 5
- 验证 NATS 投递成功但前端查询仍只依赖数据库
- 验证多 worker 并发消费与重复投递处理

## 推荐执行顺序
1. 先完成阶段 1，稳定数据库模型与 4 个接口契约。
2. 再完成阶段 2，打通 Python worker 的最小闭环。
3. 然后完成阶段 3，加固可靠性。
4. 再完成阶段 4，让前端真正消费任务结果。
5. 最后根据规模决定是否进入阶段 5，引入 NATS JetStream。
