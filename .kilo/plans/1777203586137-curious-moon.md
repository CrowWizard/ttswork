# 轻量多模块工作区改造计划

## 背景与现状
- 当前首页路由 `app/page.tsx` 仅渲染 `components/voice-studio.tsx`。
- 现有语音工作台主体结构已经拆成 `WorkspaceHeader`、`RecordingPanel`、`TtsPanel`，满足“首页保持不大改”的前提。
- 页面视觉基元主要复用 `app-card`、`app-panel`、`app-input`、`app-button-*`，适合继续做低干扰任务型 UI。
- `DESIGN.md` 当前在 Navigation 章节明确写着“当前没有全局导航”，需要做最小必要修订。
- `app/layout.tsx` 目前是全站根布局。如果把导航加到这里，会误伤 `register`、`settings` 等非主模块页，不符合需求。

## 路由确认\n- 已确认 B站视频分析页面使用 `/video-analysis`。\n- 先前出现在建议目录与验收标准中的 `/bilibili-analysis` 按笔误处理，实施时统一采用 `/video-analysis`。

## 方案对比

### 方案 A：在两个主模块页面内各自挂载共享导航组件（推荐）
- 做法：新增 `WorkspaceModuleNav`，分别在 `VoiceStudio` 首页和新的视频分析页顶部引入。
- 优点：
  - 首页改动最小，只是在 `WorkspaceHeader` 上方插入一个轻量切换条。
  - 不会影响 `register`、`settings` 等次级页面。
  - 语义最贴近“工作区模块切换”，而不是全站导航。
  - 后续新增第 3 个主模块时，仍可按同样模式扩展。
- 缺点：
  - 两个主模块页都要手动包一层相似的外部容器。

### 方案 B：新增专用工作区布局，仅给主模块路由组使用
- 做法：把 `/` 与 `/video-analysis` 放进同一个 route group/layout，由 layout 统一渲染导航。
- 优点：
  - 共享结构更集中。
  - 后续模块增多时布局更统一。
- 缺点：
  - 需要调整首页目录结构，改动面更大。
  - 对当前简单项目来说偏重，不符合“不要做与任务无关的大规模重构”。

### 推荐结论
- 采用方案 A。
- 理由：能最大限度保持首页结构不变，同时避免把“模块切换条”误实现成全站导航系统。

## 拟修改/新增文件
- `components/workspace-module-nav.tsx`
  - 新增轻量模块切换组件。
- `components/voice-studio.tsx`
  - 在现有主体顶部插入 `WorkspaceModuleNav`，其余结构尽量不动。
- `app/video-analysis/page.tsx` 或 `app/bilibili-analysis/page.tsx`
  - 新增 B 站视频分析页面入口。
- `components/bilibili-analysis/bilibili-analysis-workspace.tsx`
  - 新增页面主体骨架组件。
- `app/layout.tsx`
  - 视实现情况决定是否最小更新 `metadata` 文案；如不影响需求可不动。
- `app/globals.css`
  - 仅在现有基元不足时补充极少量模块导航样式；优先直接复用 Tailwind + 现有 token，不额外抽复杂样式。
- `DESIGN.md`
  - 最小必要更新 Navigation 描述。
- `operations-log.md`
  - 若仓库中已有该文件则追加记录；若实际不存在，实施时先确认当前项目的既有记录方式，避免无根据新增无关文档。

## 详细实施步骤

### 1. 新增轻量模块切换组件
- 组件名：`WorkspaceModuleNav`。
- 建议实现：客户端组件，使用 `usePathname()` 判定当前路由。
- 结构：仅两个 `Link`。
  - `语音工作台` -> `/`
  - `B站视频分析` -> `/video-analysis`（待用户确认是否改为 `/bilibili-analysis`）
- 交互与样式要求：
  - 整体采用 segmented tabs / pills 视觉。
  - 容器使用 `surface-muted` / `surface-elevated` / `border-subtle` 体系。
  - 当前项有更强前景层级，如 `bg-surface-elevated` + `text-text-primary` + 细阴影。
  - 非当前项保持低干扰，但 hover 明确。
  - 移动端允许横向铺满或自然换行，不引入复杂滚动容器，保证 44px 触控高度。
- 组织原则：
  - 只负责模块切换，不混入设置、帮助、登录、历史等次级入口。

### 2. 首页最小改动接入导航
- 仅修改 `components/voice-studio.tsx`。
- 在现有 `max-w-5xl` 容器内，把 `WorkspaceModuleNav` 放在 `WorkspaceHeader` 之前。
- 保持以下部分不重构：
  - `WorkspaceHeader`
  - `RecordingPanel`
  - `TtsPanel`
  - 登录弹窗逻辑
- 预期改动范围：
  - 只新增一个 import。
  - 只在主内容顶部插入一个导航块。
  - 原双栏 section 与间距做最小必要微调。

### 3. 新增 B 站视频分析独立页面
- 新建页面路由：`app/video-analysis/page.tsx`。
- 页面主体建议抽成 `components/bilibili-analysis/bilibili-analysis-workspace.tsx`，便于以后接 API 时局部演进。
- 页面结构：
  - 顶部：`WorkspaceModuleNav`
  - 主标题卡：
    - 标题：`B站视频分析`
    - 说明：`输入视频链接或 BV 号，生成摘要、结构拆解与可复用文案。`
  - 输入区卡片：
    - 文本输入框：支持视频链接 / BV 号
    - `开始分析` 主按钮
    - 简短说明文字：例如支持输入完整链接或 `BV...` 编号
  - 结果区卡片：
    - 未分析空态
    - 预留静态分区：
      - 视频基础信息
      - 内容摘要
      - 分段结构
      - 金句 / 爆点
      - 可复用文案建议
- 页面布局建议：
  - 不复用首页左右双栏。
  - 用单列纵向工作流，更符合“输入 -> 结果”的任务链路。
  - 在宽屏下可让结果区内部做 2 列摘要布局，但整体页面仍保持单主轴，不做后台控制台感。

### 4. 结果区空态与骨架约定
- 未分析时，结果区直接展示明确空态，而不是空白占位。
- 空态文案方向：
  - 说明先输入链接或 BV 号。
  - 告知分析后将输出摘要、结构拆解和文案建议。
- 静态骨架阶段不接后端，不请求接口，不引入 mock service。
- 可以用以下轻量占位方式：
  - 每个结果分区使用标题 + 1 段说明文字。
  - 或者用淡色占位块提示未来会承载的内容。

### 5. 最小更新 DESIGN.md
- 仅更新 Navigation 相关描述，避免大范围重写文档。
- 目标表述应改为：
  - 当前采用轻量任务导航
  - 仅承载核心工作区模块切换
  - 不扩展为营销导航或后台导航
- 如有必要，补一句当前入口仅包含“语音工作台 / B站视频分析”两个主模块。

### 6. 验证步骤
- 运行 `npm run lint`
- 运行 `npm run typecheck`
- 运行 `npm run build`
- 重点检查：
  - `/` 首页仍保留原有语音工作台主体结构。
  - `/video-analysis` 页面正常打开。
  - 导航在两个模块页都能正确高亮。
  - 移动端断点下导航与输入卡片不溢出。
  - `register`、`settings` 页面未被强行加上主模块导航。

## 实施时的具体注意点
- 不修改后端代码。
- 不把模块导航塞进 `app/layout.tsx`，否则会把非主模块页也一起改掉。
- 不新增侧边栏、全局 header、营销导航、账户中心式入口。
- 复用现有 Tailwind token 与 `app-*` 基元，避免引入新 UI 框架。
- 若组件需要注释，注释使用中文，且只解释意图。
