# 阶段 2：Python Worker 最小闭环实施计划

## 目标
- 在不引入 MQ 的前提下，基于现有 `Postgres + job table` 设计，完成视频分析异步任务的最小可运行闭环。
- 让 `POST /api/video-analysis/jobs` 创建出的 `VideoAnalysisJob` 能被独立 Python worker 领取、处理并回写结果。
- 保持数据库为任务状态真相源，前端继续通过现有 4 个接口查状态和结果。

## 当前基线
- 阶段 1 已完成：
  - `VideoSource` 与 `VideoAnalysisJob` 已入 Prisma schema。
  - 4 个 `/api/video-analysis` 接口骨架已存在。
  - `POST /jobs` 已能解析 B 站链接/BV，复用或创建 `VideoSource`，并新建 `VideoAnalysisJob`。
  - `GET` 接口已经通过 DTO 返回数据库内容。
- 现有实现位置：
  - 路由：`api-server/src/routes/video-analysis.ts`
  - 路由挂载：`api-server/src/index.ts:138`
  - 参数校验：`api-server/src/lib/validation.ts:56`
- 当前缺口：
  - 没有 worker 领取任务。
  - `VideoSource` 没有被补齐视频元信息、字幕、转写文本。
  - `VideoAnalysisJob` 没有从 `PENDING` 流转到 `READY/FAILED`。

## 约束与边界
- 只支持登录用户，不接入 `anonymousUserId`。
- 继续采用弱缓存：
  - `VideoSource` 按 `normalizedBvid` 复用。
  - `VideoAnalysisJob` 每次都新建。
- 当前阶段不做：
  - NATS / Dapr / Celery
  - 任务锁超时回收与重试策略的完整加固
  - 前端轮询接入
- 当前阶段可以接受：
  - 单 worker 进程常驻轮询数据库
  - 单任务串行或低并发处理
  - 结果结构先严格对齐现有页面 5 个分区

## 推荐落地方案

### 方案 A：新建独立 `video-analysis-worker/` Python 子目录（推荐）
- 在仓库根目录新增独立 Python worker 目录，例如：
  - `video-analysis-worker/`
- 目录内独立维护：
  - `requirements.txt` 或 `pyproject.toml`
  - worker 入口脚本
  - 配置加载
  - 数据库访问
  - B 站抓取/字幕/ASR/LLM 模块
- 优点：
  - 与 `api-server/` 职责边界清晰
  - 不污染现有 Bun / TS 工程
  - 部署与调试时更容易独立运行
- 缺点：
  - 仓库会新增一套 Python 依赖管理

### 方案 B：将 Python worker 放到 `scripts/` 下
- 例如：`scripts/video_analysis_worker/`
- 优点：
  - 新文件少一点
- 缺点：
  - 随着业务增长，目录语义会变混乱
  - 不利于后续把 worker 单独部署

### 推荐结论
- 采用方案 A，新建独立 `video-analysis-worker/`。
- 理由：Python worker 会逐渐演进成一个独立长期存在的执行单元，不适合塞进 `scripts/`。

## 阶段 2 的实现步骤

### 步骤 1：固定 Python worker 的配置来源
目标：让 worker 能稳定读取与 API 相同的基础配置。

建议读取来源：
1. 根目录 `.env`
2. 可选单独配置文件，如 `video-analysis-worker/.env`
3. 若需要 YAML，再额外支持 `api-server/config.yaml`

最小必需配置：
- `DB_HOST`
- `DB_PORT`
- `DB_NAME`
- `DB_USER`
- `DB_PASSWORD`
- `DB_SCHEMA`
- `LOG_LEVEL`
- `QWEN_API_KEY` 或视频分析使用的 LLM Key
- B 站抓取相关配置（如 Cookie / UA，若需要）
- ASR 相关配置

建议：
- 第一版尽量与 `api-server/src/lib/config.ts` 对齐同名环境变量，避免出现两套命名。

### 步骤 2：实现数据库访问层
目标：让 worker 能安全领取任务、读取源数据、回写结果。

需要的最小 SQL 能力：
1. 领取 1 条 `PENDING` 任务
2. 原子更新任务为 `PROCESSING`
3. 按 `videoSourceId` 读取 `VideoSource`
4. 更新 `VideoSource` 的元信息 / 字幕 / 转写状态
5. 更新 `VideoAnalysisJob` 的结果与最终状态

推荐领取策略：
- 先查一条最早的 `PENDING` 任务
- 在事务内执行条件更新：
  - 仅当 `status = 'PENDING'` 时，更新为 `PROCESSING`
  - 同时写入 `workerId` 与 `lockedAt`
- 若更新行数为 0，则说明被其他 worker 抢走，继续下一轮

注意：
- 这是阶段 2，允许简单轮询
- 不必在本阶段引入复杂的心跳、租约续约和抢锁恢复

### 步骤 3：实现 `VideoSource` 内容补齐链路
目标：根据 `normalizedBvid` 完成视频基础信息与文本来源准备。

顺序建议：
1. 读取 `VideoSource.normalizedBvid`
2. 拉取视频基础信息：
  - `title`
  - `authorName`
  - `authorMid`
  - `coverUrl`
  - `durationSeconds`
  - `publishTime`
3. 尝试抓字幕
4. 若抓到字幕：
  - `subtitleStatus = READY`
  - `subtitleText = ...`
  - `transcriptStatus = READY`
  - `transcriptSource = SUBTITLE`
  - `transcriptText = subtitleText`
5. 若字幕不存在：
  - `subtitleStatus = UNAVAILABLE`
  - 下载或提取音频后做 ASR
  - `transcriptStatus = READY`
  - `transcriptSource = ASR`
  - `transcriptText = asrText`
6. 若任一步失败：
  - 回写 `fetchErrorMessage`
  - 视失败阶段更新 `subtitleStatus` / `transcriptStatus`

建议：
- 如果 `VideoSource` 已经存在且 `transcriptText` 不为空，worker 可直接复用，不必重复抓字幕/跑 ASR。
- 这正好体现弱缓存价值。

### 步骤 4：实现字幕优先策略
目标：保证“有字幕优先用字幕，无字幕才走 ASR”。

明确策略：
- 优先级固定为：`subtitle > asr`
- 不允许在已有字幕时仍强制跑 ASR
- `transcriptSource` 必须准确写明来源

需要明确的失败分类：
- 字幕不存在：不是错误，属于 `UNAVAILABLE`
- 字幕抓取异常：属于 `FAILED`
- ASR 执行异常：属于 `FAILED`

### 步骤 5：实现结构化分析器
目标：把 `transcriptText` 转成页面可渲染的结构化结果。

输出必须严格对齐当前页面分区：
1. `summary`
2. `structureSections`
3. `highlights`
4. `copySuggestions`

推荐输出格式：
- `summary`: `string`
- `structureSections`: `[{ title, startSeconds, endSeconds, summary }]`
- `highlights`: `[{ quote, reason, timestampSeconds }]`
- `copySuggestions`: `[{ type, content }]`

建议做法：
- LLM 输出 JSON
- Python 端先做 schema 校验
- 校验通过后再回写数据库
- 若 LLM 返回非 JSON 或字段缺失：
  - 当前任务记为 `FAILED`
  - `errorMessage` 写入“分析结果格式不合法”之类的稳定文案

建议 prompt 约束：
- 明确要求输出合法 JSON
- 明确字段名和数组结构
- 明确禁止额外文本包裹

### 步骤 6：回写任务状态
目标：保证前端查到的任务状态和结果一致。

成功时：
- `VideoAnalysisJob.status = READY`
- 写入：
  - `summary`
  - `structureSections`
  - `highlights`
  - `copySuggestions`
  - `modelName`
  - `promptVersion`
  - `completedAt`
  - `errorMessage = null`

失败时：
- `VideoAnalysisJob.status = FAILED`
- 写入：
  - `errorMessage`
  - 可保留 `workerId`
  - 可写 `completedAt`

注意：
- 当前阶段不要在失败时自动重试
- `retryCount` / `nextRetryAt` 留给阶段 3

## 建议的任务状态流转

### 成功路径
`PENDING -> PROCESSING -> READY`

### 失败路径
`PENDING -> PROCESSING -> FAILED`

### 当前阶段不实现
- `FAILED -> PENDING` 自动重试
- `PROCESSING` 超时回收

## 需要明确的实现决策

### 1. Python 数据库访问库
候选：
1. `psycopg`（推荐）
2. `asyncpg`
3. SQLAlchemy

推荐：`psycopg`
- 当前阶段同步轮询足够
- 事务与条件更新清晰
- 依赖更轻

### 2. B 站信息抓取实现
候选：
1. 官方公开接口 / 网页解析
2. 第三方库

推荐：
- 优先选稳定、依赖少的实现
- 尽量封装在 `services/bilibili.py`
- 避免把站点解析逻辑分散到 worker 主流程中

### 3. ASR 提供方
候选：
1. 复用现有 Qwen 体系
2. 独立 ASR 服务
3. 本地模型

推荐：
- 阶段 2 先选接入成本最低、当前环境最容易跑通的方案
- 计划实现时要把 ASR 抽象成单独服务接口，不把供应商细节写死在 worker 主流程

### 4. LLM 分析结果校验
推荐：
- 用 Python `pydantic` 或手写 schema 校验 JSON
- 不要把未经校验的模型输出直接写入数据库

## 验证计划

2. API 侧验证
- 登录后调用 `POST /api/video-analysis/jobs`

3. Worker 验证
- 启动 Python worker
- 观察任务从：
  - `PENDING`
  - 到 `PROCESSING`
  - 到 `READY` 或 `FAILED`

4. 详情接口验证
- 调用 `GET /api/video-analysis/jobs/:jobId`
- 确认：
  - `source.title` 已填充
  - `source.transcriptSource` 正确
  - `result.summary` / `structureSections` / `highlights` / `copySuggestions` 有值

5. 两类样本验证
- 一条有字幕视频
- 一条无字幕视频

### 重点检查项
- 同一 `normalizedBvid` 再次提交任务时，是否复用 `VideoSource`
- 同一视频再次分析时，是否新建 `VideoAnalysisJob`
- 字幕存在时，是否没有错误进入 ASR
- 分析输出是否严格符合页面结构要求

## 风险与规避

### 风险 1：B 站字幕接口不稳定
规避：
- 在 `services/subtitle.py` 封装单一入口
- 把“无字幕”和“字幕抓取失败”区分开

### 风险 2：ASR 耗时过长
规避：
- 阶段 2 先接受串行长任务
- 不在阶段 2 解决并发吞吐问题

### 风险 3：LLM 输出不稳定
规避：
- 严格限制 JSON schema
- 失败时直接标记任务失败，不写脏数据

### 风险 4：多 worker 重复领取
规避：
- 必须使用事务 + 条件更新抢任务
- 即使当前只打算跑 1 个 worker，也不能省略这层保护

## 验收标准
- 能新建并运行至少 1 条公开视频分析任务
- 有字幕视频成功走字幕链路
- 无字幕视频成功走 ASR 链路
- `GET /api/video-analysis/jobs/:jobId` 返回完整结构化结果
- 同一视频重复分析时，复用旧 `VideoSource`，但新增 `VideoAnalysisJob`
- 失败任务能稳定落到 `FAILED`，并带 `errorMessage`
